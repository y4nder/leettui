<task type="auto">
  <name>Task 1: Refresh the History panel after a live submit (Gap 1, HIST-03)</name>
  <files>src/views/problem/handlers.ts</files>
  <action>
In `handleProblemSubmit`'s `withFocusedSolution` success callback — currently it awaits `submitSolution(p.question, langSlug)` into `result` and then calls only `useAppStore.getState().setProblemResult(buildResultView(result))` — add a call to `useAppStore.getState().loadProblemSubmissions(p.question.id)` immediately after `submitSolution` resolves and before the `setProblemResult` call, so the History panel and the Result panel update together.

Why this is correct and sufficient: `submitSolution` (core/submission.ts) has already persisted the new submission row to SQLite via `appendSubmissionRecord`/`insertSubmission` — for every verdict, not just Accepted — before its promise resolves, so re-reading `getSubmissionsForQuestion` at this point is guaranteed to see the fresh row. `loadProblemSubmissions` is the sole action that repopulates `submissionsSlice`'s `problemSubmissions`/`submissionSummary`, and `ProblemView` renders `HistoryPanel` from those via reactive selectors, so re-invoking it forces the in-session re-render. `p` (the problem) is already bound in this callback's scope, so `p.question.id` is in scope. This mirrors the refresh-after-mutation pattern already used in `handleConfirmDeleteSolution` (which calls `refreshProblemSolutions()` / `refreshSolutionFiles()` after `deleteSolution`).

Scope guards:
- Do NOT add the same call to `handleProblemRun`: `runSolution` never calls `appendSubmissionRecord` (only submit persists a submission row, per the debug session), so Run has nothing to refresh.
- Do NOT change `handleProblemSubmit`'s structure, its `withFocusedSolution` options (`loadingMessage` / `errorTitle` / `solutionErrorArm`), or its error handling.
- Do NOT touch the browse attempt-count badge here — it already recomputes on `handleExitProblemView`'s `refreshQuestions(...)` call; this task is only the in-session History-panel refresh.
  </action>

  </verify>

<task type="auto">
  <name>Task 2: Make the browse attempt-count badge fixed-width (Gap 2, BROWSE-01)</name>
  <files>src/ui/components/QuestionList.tsx</files>
  <action>
The per-row marker cell renders a variable-length string (`×${attemptCount}` when `attemptCount > 0`, else `◆` when a solution file exists, else a single space) directly inside its `<text fg={colors.accent}>` with no padding — unlike its sibling data columns `idStr`, `acStr`, and `runtimeStr`, which are each pre-padded to a fixed character count with `.padStart()` in the per-row computation block (the block that currently defines `idStr = String(q.id).padStart(4, " ")`, `acStr = ...padStart(6, " ")`, `runtimeStr = ...padStart(8, " ")`). Because the row is a `flexDirection="row"` box with no explicit width on the marker `<text>`, a wider badge (e.g. `×32`) pushes every following column right, breaking vertical alignment.

Fix: introduce a new `markerStr` const in that same per-row computation block (alongside `idStr`/`acStr`/`runtimeStr`), holding the existing D-14 precedence value and right-aligned via `.padStart` to a fixed width of 3 (single space fill). Width 3 right-aligns `◆`, the blank fallback, and `×1`–`×9` within a slot that also exactly fits `×99` — the realistic ceiling for one problem's total attempts across all verdicts; the sibling columns use the same `.padStart` idiom. (A hypothetical `×100+` overflows the slot by one column — an acceptable extreme edge, not worth widening the common blank/`◆` rows further.) Then render the marker cell as the accent-colored `<text>` interpolating `markerStr` in place of the current inline conditional expression.

Scope guards:
- Keep the marker in the SAME slot/column: the existing accent-colored `<text>` sitting between the status-icon cell and the ` {idStr} ` cell. Do NOT add a new column, move any sibling column, or alter `idStr`/`acStr`/`runtimeStr` (D-13 — the badge reuses the ◆ slot without moving neighbors).
- Preserve the D-14 precedence exactly: `×N` when `attemptCount > 0`, else `◆` when a solution file exists (`hasSolution`), else the blank fallback.
  </action>
  <verify>
    <automated>bun run typecheck && bun run lint (both pass); then `grep -q 'markerStr' src/ui/components/QuestionList.tsx` and `grep -q 'padStart(3' src/ui/components/QuestionList.tsx` both succeed (the marker cell is now pre-padded to a fixed width like its siblings, taking QuestionList's total `.padStart` count from 3 to 4).</automated>
    <human-check>In browse, view the question list where rows have mixed badges (a `×1`, a `×32`, a `◆`, and a blank/never-attempted row); confirm the problem-number, title, acceptance-rate, difficulty, and runtime columns line up vertically across all rows regardless of badge digit count.</human-check>
  </verify>
  <done>The marker slot is fixed-width (right-aligned `.padStart(3)`); browse columns stay aligned regardless of `×N` digit count; the D-13 slot position and D-14 precedence are unchanged; the full pre-commit gate is green.</done>
</task>
