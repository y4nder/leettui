# Phase 02.2 Plan 01: Capture Engine Summary

**Pure, dir-injected, never-throws capture engine (`captureFailingCase` + `blessRunOutputs`) that turns a LeetCode `CheckResponse` into gradeable local `tests/case-NN.txt`/`.out` files, proven by a round-trip test against `discoverCases`/`compareOutput`**

## Accomplishments
- `captureFailingCase(testsDir, input, expectedOutput?)` implements all six D-01/D-03/D-04/D-05/D-06 branches: new-case capture (input+expected or input-only), normalized-duplicate detection, missing-`.out` fill, and disagreeing-`.out` mismatch surfacing (never overwritten)
- `blessRunOutputs(testsDir, exampleTestcaseList, expectedCodeAnswer?)` fills missing `.out`s for existing seeded cases by content match (never index), skipping already-blessed cases and unmatched answers
- Round-trip proven: a case captured by `captureFailingCase` is immediately discoverable with a non-null `expectedPath` and grades `pass` via `compareOutput` — the core offline-gradeability promise of the phase
- Path-safety invariant (T-02.2-01) asserted by a dedicated test: a `../../etc/x`-style input still lands as file content under a generated `case-NN` name, never as a path segment
- `normalize` exported from `testRunner.ts` (single shared normalization rule, no duplication)
- TCASE-04/TCASE-05 formalized in `REQUIREMENTS.md` with traceability rows and updated coverage counts (23 total / 23 mapped)
- `src/core/solutions/index.ts` barrel re-exports `captureFailingCase`, `blessRunOutputs`, `CaptureOutcome`, `CaptureResult` for Plan 02's `submission.ts` hook
