# Phase 02.2 Plan 02: Wire Capture into Submission Service + Result Rendering Summary

**`runSolution`/`submitSolution` now return `{ response, captureNotes }` — a Pitfall-5-guarded service hook that turns a real LeetCode run/submit into an auto-captured or blessed offline test case, surfaced as a dim confirmation line in both the TUI (inline panel, fullscreen takeover, browse popup) and the headless CLI**

## Accomplishments
- `submission.ts`'s `runSolution`/`submitSolution` now return `{ response: ParsedResponse; captureNotes: string[] }`, with the capture call wrapped in its own narrow `try`/`catch` (defense in depth on top of the Plan 01 writers' own never-throws contract — Pitfall 5 / T-02.2-04) so a capture regression can never turn a successful run/submit into an error screen
- `captureFromSubmitResult(testsDir, result)` branches on the already-parsed `ParsedResponse.type` (no new response parsing): `submit_wrong_answer` captures input+expected (D-01), `runtime_error`/`time_limit_exceeded`/`memory_limit_exceeded` capture input-only (D-03, `expected_output` never substituted with a placeholder — Pitfall 1), everything else captures nothing
- `captureFromRunResult(testsDir, exampleTestcaseList, result)` blesses missing `.out`s on any completed run (`run_accepted` or `run_wrong_answer`, D-02) via the Plan 01 `blessRunOutputs` writer, content-matched never index-matched (D-07)
- Both helpers are `testsDir`-injected (not internally resolving `getTestsDir`), mirroring every other writer in `core/solutions/` — directly unit-testable against a real temp dir with zero config/network mocking, unlike the RESEARCH.md design sketch's internally-resolving shape
- All four production call sites updated in the same commit so `tsc --noEmit` stays green throughout (Pitfall 2): `handleProblemRun`/`handleProblemSubmit` (TUI) and `runApiVerb`'s shared `action` type (CLI `run`/`submit`)
- `ResultView` gained `notes?: string[]`; `ResultBody.tsx` renders it as dim lines after `error` and before `cases` — reaching the inline Result panel, `ResultFullscreen` takeover, and `ResultPopup` at once since they all share the one component; `presentResultView` mirrors it with `c.dim`, leaving `exitCodeForResultView` untouched (capture notes never affect verdict color or exit code)
- 11 new unit tests in `submission.test.ts` cover both helpers across all D-01–D-07 branches plus the Pitfall-5 degrade-to-`[]` case

## Decisions Made
- `captureFromSubmitResult`/`captureFromRunResult` take `testsDir: string` directly instead of resolving `getTestsDir(question.id, question.title_slug)` internally (as RESEARCH.md's design sketch did) — this keeps them pure and config-free, consistent with `capture.ts`'s own dir-injected shape (Pattern 1 from RESEARCH/PATTERNS), and lets `submission.test.ts` exercise them against a real temp dir with zero config/network mocking. `submission.ts` resolves `testsDir` once per call site and passes it in; the D-08 hook point and the try/catch placement match the plan unchanged.
- Run-blessing reports a single summary note (`"Blessed N expected output(s)"`) rather than enumerating per-case, per RESEARCH.md's Open Question 1 recommendation for the common case — the underlying `blessRunOutputs` still returns per-case `CaptureResult[]`, so a future caller could enumerate if needed.
