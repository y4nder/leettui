## User Constraints (from CONTEXT.md)

### Locked Decisions

**Capture triggers (which verdicts write files)**
- **D-01:** Submit wrong-answer is the headline capture: `last_testcase` → new `case-NN.txt`, `expected_output` → sibling `case-NN.out` (subject to dedup rules). The run path also participates: run responses bless missing `.out`s for existing seeded cases from `expected_code_answer[]` — authoritative data, better than `--save`'s trust-the-user model.
- **D-02:** Run-blessing fires on any completed run (accepted *or* wrong-answer) — `expected_code_answer[]` is LeetCode-authoritative regardless of the user's verdict; an accepted run is in fact the safest blessing moment.
- **D-03:** RE/TLE/MLE submits capture input-only: `last_testcase` (present without `expected_output` on these verdicts) is written as `case-NN.txt` with no `.out` — offline `leettui test` immediately reproduces the crash/timeout (status `error`/`timeout`), and the `.out` is blessed later by a run or `--save` once fixed. Compile-error / pending / timeout / unknown responses carry no testcase and never capture.

**Dedup & conflict rules (non-destructive invariant)**
- **D-04:** "Already captured" = normalized input match against every existing `case-NN.txt`: trailing-whitespace/newline + CRLF-insensitive comparison (same spirit as `compareOutput`'s lenient string fallback). Catches cosmetic-whitespace dupes without false positives.
- **D-05:** If the failing input matches an existing case that has no sibling `.out` and the response carries `expected_output`, fill the missing `.out` for that existing case instead of writing a duplicate — a fill, not a clobber; the case becomes gradeable.
- **D-06:** If the matching case already has a `.out` that disagrees with LeetCode's expected output, never overwrite it — surface a mismatch note (e.g. "case-03.out disagrees with LeetCode's expected output") so the user re-blesses deliberately via `test --save` or hand-edit. The auto path is strictly additive (ROADMAP success criterion 2).
- **D-07:** Run-blessing maps `expected_code_answer[i]` to the local case whose `.txt` content matches `exampleTestcaseList[i]` (match by input content, same normalization as D-04) — never by index/position, since the local `tests/` dir may hold user-added (`--add-case`) or 2.2-captured cases that weren't part of the run. An unmatched answer is simply not written.

**Hook placement & surfaces**
- **D-08:** Capture hooks inside the core submission service — `runSolution`/`submitSolution` in `src/core/submission.ts` invoke the capture writer (which lives in `src/core/solutions/` beside `addCase`/`saveGoldenOutputs`) and return capture info alongside the `ParsedResponse`. The TUI (`R`/`s`) and the headless CLI (`leettui run`/`submit`) both get capture for free and can never drift — consistent with the service already doing side effects (`appendSubmissionRecord`, `markAttempted`).

**Defaults adopted without explicit confirmation** *(user went AFK for the last two questions; recommended options adopted — override by editing CONTEXT.md before planning if wrong)*
- **D-09 (adopted by default):** Always-on, no config knob. Capture is additive-only, never overwrites, and degrades silently — nothing harmful to opt out of; matches leettui's lean/YAGNI philosophy. A `[test]` toggle can be added later if someone asks.
- **D-10 (adopted by default):** Capture feedback = line(s) in the existing result view (TUI fullscreen takeover / Result panel, and the CLI's `presentResultView`): e.g. "Captured tests/case-05 (input + expected)", "Blessed 2 expected outputs", and the D-06 mismatch note. Reuses the existing view-builder seam (`views/browse/resultView/builders.ts`, `cli/present.ts`); no new UI surface.

## Summary

This phase is a pure internal-TypeScript extension — no new packages, no new external services. It adds one filesystem side effect to an already-existing, already-authenticated flow (`runSolution`/`submitSolution` in `src/core/submission.ts`), reusing three seams Phase 2.1 already built: `nextCaseName`/`addCase` (sequential case-input writing), `saveGoldenOutputs`'s outcome-reporting model (created/changed/unchanged/skipped), and `compareOutput`'s normalization philosophy (trailing-whitespace/CRLF-insensitive, JSON-canonicalized fallback). The new work is two writer functions — a submit-side "capture a new failing case" writer and a run-side "bless missing `.out`s by content match" writer — plus threading their outcome back through two call sites (`views/problem/handlers.ts`'s `handleProblemRun`/`handleProblemSubmit`, and `cli/index.ts`'s `runApiVerb`) into two render surfaces (`ResultBody.tsx` for the TUI, `presentResultView` in `cli/present.ts` for the CLI) that already share one `ResultView` type.

The single largest design decision left open by CONTEXT.md is **how capture info threads back to callers** (explicitly "Claude's discretion"). `runSolution`/`submitSolution` currently return `Promise<ParsedResponse>` and are called from exactly two production call sites (verified by grep — `views/problem/handlers.ts` lines ~323/351, `cli/index.ts` lines ~168/175 via `runApiVerb`'s `action` callback). Changing the return shape to `{ response: ParsedResponse; capture: CaptureNote[] }` (or similar) is a small, fully-enumerable blast radius, and the natural rendering seam is a new optional `notes?: string[]` field on `ResultView` (rendered by `ResultBody.tsx`, which `ResultFullscreen.tsx` and `ProblemView.tsx` already consume, and by `presentResultView` in `cli/present.ts`) — one field addition propagates to every render surface for free, matching D-10's "reuse the existing view-builder seam" instruction.

**Primary recommendation:** Add a dir-injected, pure, never-throws capture module `src/core/solutions/capture.ts` (mirroring `add-case.ts`/`save-output.ts`'s shape exactly — `(testsDir: string, ...) => CaptureResult[]`), export `normalize` from `testRunner.ts` for reuse (D-04/D-07 need the same trailing-whitespace/CRLF-insensitive comparison `compareOutput` already has internally but doesn't expose), hook both writer functions into `runSolution`/`submitSolution`, change their return type to carry capture notes, add `notes?: string[]` to `ResultView`, and render it in both `ResultBody.tsx` and `presentResultView`.

### Pattern 1: Dir-injected, pure, never-throws writer (established by Phase 2.1)
**What:** Every case-writing function takes `testsDir: string` explicitly (never resolves `getTestsDir(id, slug)` itself), performs synchronous `node:fs` calls with no try/catch (the codebase's stated reasoning: `mkdirSync`/`readFileSync`/`writeFileSync` on a resolvable path don't throw in practice), and returns a small outcome type rather than throwing.
**When to use:** Both new capture functions (`captureFailingCase`, `blessRunOutputs`).
**Example:**
```typescript
// Source: src/core/solutions/save-output.ts (existing pattern to mirror)
export function saveGoldenOutputs(testsDir: string, cases: LocalCaseResult[]): SaveResult[] {
  return cases.map((c): SaveResult => {
    if (c.status === "error" || c.status === "timeout" || c.actual === undefined) {
      return { name: c.name, outcome: "skipped" };
    }
    mkdirSync(testsDir, { recursive: true });
    const path = join(testsDir, `${c.name}.out`);
    const existed = existsSync(path);
    const prev = existed ? readFileSync(path, "utf-8") : null;
    writeFileSync(path, c.actual);
    if (!existed) return { name: c.name, outcome: "created" };
    return { name: c.name, outcome: compareOutput(c.actual, prev as string) ? "unchanged" : "changed" };
  });
}
```

### Pattern 2: Service-layer side effect, invoked unconditionally after the poll resolves
**What:** `submitSolution` already does `appendSubmissionRecord(...)` + `markAccepted`/`markAttempted` as side effects after `pollResult` resolves, before returning. Capture is one more such side effect at the same point — D-08 explicitly calls this out as the reason to hook here rather than in the view layer, so TUI and CLI can never drift.
**When to use:** Both `runSolution` and `submitSolution`.
**Example:**
```typescript
// Source: src/core/submission.ts (existing pattern to extend)

  const code = loadCode(question, langSlug);
  const editorData = await fetchEditorData(question.title_slug);

    markAttempted(question.id);
  }

}
```

### Pattern 3: Content-based matching, never index/position-based (D-04, D-07)
**What:** `tests/` may hold user-added (`--add-case`) or previously-captured cases interleaved with the originally-seeded example cases, so `exampleTestcaseList[i]` cannot be assumed to correspond to `case-{i+1}`. Both the submit-side dedup (D-04) and the run-side blessing match (D-07) key off **normalized input content**, exactly as `compareOutput` already does for output grading.
**When to use:** Any lookup from a `CheckResponse` field back into the local `tests/` dir.
**Example (design sketch — not yet in the codebase):**
```typescript
// capture.ts design sketch
import { normalize } from "../testRunner"; // requires exporting `normalize` (currently private)
import { discoverCases } from "../testRunner";
import { readFileSync } from "node:fs";

function findCaseByInput(testsDir: string, input: string): DiscoveredCase | undefined {
  const target = normalize(input);
  return discoverCases(testsDir).find((c) => normalize(readFileSync(c.inputPath, "utf-8")) === target);
}
```
Why this must not be index-based: `seedTests` numbers `case-${i+1}` from `exampleTestcaseList` **only at creation time** (verified in `src/core/solutions/create.ts`); any later `--add-case` (Phase 2.1) or 2.2 capture appends higher numbers without regard to `exampleTestcaseList` order, and a user can delete/reorder manually. A subsequent `run`'s `exampleTestcaseList[i] ↔ case-(i+1)` assumption would silently bless the wrong case.

### Anti-Patterns to Avoid
- **Index-matching `expected_code_answer[i]` to `case-(i+1).out`:** Breaks the moment any case has been added or reordered since the problem was created — exactly the scenario 2.1 and 2.2 both introduce. Always content-match (D-07).
- **Throwing from the capture writer:** Violates ROADMAP success criterion 4 ("capture never destabilizes the run/submit path") and the established never-throws contract shared by `seedTests`/`addCase`/`saveGoldenOutputs`/`git.ts`. A parse/write failure must degrade to "no capture, note omitted," never propagate.
- **Re-deriving verdict logic already in `parseCheckResponse`:** The design note explicitly says "no new response parsing" — capture branches on the already-discriminated `ParsedResponse.type`, not on raw `status_code`.

| Problem | Don't Build | Use Instead | Why |

| Trailing-whitespace/CRLF-insensitive string comparison | A second, subtly different `normalize()` in `capture.ts` | Export and reuse `normalize` from `src/core/testRunner.ts` | `compareOutput` already solved this for output-grading; D-04/D-07 need the *identical* rule for input-matching. Two independent implementations risk silent divergence on a later edit to one but not the other. |

| `.out` write outcome bookkeeping (created/changed/unchanged/skipped) | A new bespoke result type for capture | Model `CaptureResult`/`BlessResult` on `save-output.ts`'s `SaveOutcome` union, extended with a `mismatch` arm for D-06 | Consistent reporting vocabulary across `--save`, capture, and blessing; the CLI/TUI renderers can share glyph-mapping logic (`saveOutcomeGlyph` in `cli/present.ts` is the direct precedent to extend, not replace). |
| JSON-vs-string output equality for a captured `.out` | Custom deep-equal | `compareOutput` (already JSON-canonicalizing with a string fallback) | Same tool `saveGoldenOutputs`/`testRunner.ts` already use to decide changed-vs-unchanged. |

**Key insight:** Phase 2.1 already built every primitive this phase needs (numbering, input writing, output writing, comparison). The entire scope of 2.2 is (a) two new *decision* functions that call those primitives under the D-01..D-07 branching rules, and (b) threading a small feedback string list through an existing return type and two existing render paths. Resist the urge to build a new "capture engine" abstraction — it is two functions beside two siblings.

## Common Pitfalls

### Pitfall 1: Assuming `expected_output`/`last_testcase` are always present together
**What goes wrong:** Code that unconditionally destructures both fields off `CheckResponse` and writes a `.out` will write garbage (or crash) on RE/TLE/MLE submits, where `last_testcase` is present but `expected_output` is not (D-03).
**Why it happens:** `submit_wrong_answer`'s builder (`buildResultView`) already treats `expected_output ?? "N/A"` defensively for *display*, but a capture writer must treat absence as a *branch*, not a fallback string — `"N/A"` must never be written to a `.out` file.
**How to avoid:** Branch explicitly per `ParsedResponse.type`: only `submit_wrong_answer` writes both `.txt`+`.out`; `runtime_error`/`time_limit_exceeded`/`memory_limit_exceeded` write `.txt` only, gated on `last_testcase` being present (CONTEXT's "Claude's discretion" note: a missing `last_testcase` → no capture at all, not a half-write).
**Warning signs:** A `.out` file containing the literal string `"N/A"` or `undefined`.

### Pitfall 2: Threading capture info through the return type breaks a caller silently
**What goes wrong:** `runSolution`/`submitSolution` currently return `Promise<ParsedResponse>`, consumed directly by `buildResultView(result)` at exactly 2 call sites (`views/problem/handlers.ts` lines ~323/351, `cli/index.ts` lines ~168/175 through `runApiVerb`'s `action` parameter typed `() => Promise<ParsedResponse>`). Any return-shape change must update both call sites and `runApiVerb`'s parameter type in the same commit, or TypeScript's `tsc --noEmit` pre-commit gate (zero-tolerance, per project CLAUDE.md) will fail the build.
**Why it happens:** The two call sites are in different subtrees (`views/problem/` vs `cli/`) and easy to miss one during a quick edit.
**How to avoid:** Grep-verify all call sites before changing the signature (`grep -rn "runSolution(\|submitSolution(" src`) — confirmed here to be exactly the 2 production sites plus their existing tests. `runApiVerb`'s `action` type must change alongside.
**Warning signs:** `tsc --noEmit` failures on `views/problem/handlers.ts` or `cli/index.ts` after touching `submission.ts`'s return type.

### Pitfall 3: `normalize()` is currently module-private in `testRunner.ts`
**What goes wrong:** `capture.ts` cannot `import { normalize } from "../testRunner"` today — the function exists but is not exported (verified: only `pairCases`, `compareOutput`, `discoverCases`, `runLocalTests` are exported; `normalize` and `canonicalize`/`tryCanonicalJson` are not).
**Why it happens:** It was written as an internal helper for `compareOutput` before any other module needed the same rule.
**How to avoid:** Add `normalize` to `testRunner.ts`'s export list as part of this phase's plan (a one-line change), rather than duplicating the regex/trimEnd logic in `capture.ts`.
**Warning signs:** A near-duplicate `function normalize(s: string)` appearing in two files.

### Pitfall 4: Capture feedback needs a new `ResultView` field, not an ad-hoc string append
**What goes wrong:** Concatenating capture text into `view.title` or an existing `metrics`/`error` field would corrupt the CLI's exit-code derivation (`exitCodeForResultView` switches on `view.kind`, and `presentResultView` renders `error` in red — a capture note is neither) and the TUI's color semantics (title color maps 1:1 to verdict kind).
**Why it happens:** It's tempting to reuse an existing field rather than touch the shared `ResultView` type and both renderers.
**How to avoid:** Add `notes?: string[]` to `ResultView` (`src/views/browse/resultView/types.ts`) and render it distinctly (dim, below metrics) in both `ResultBody.tsx` and `presentResultView` — one field, both surfaces, per D-10's explicit instruction to "reuse the existing view-builder seam."
**Warning signs:** Exit code or color changing for a capture-only note with no real verdict change.

### Pitfall 5: A capture exception must not turn a successful submit into an error screen
**What goes wrong:** If the capture writer is invoked with a naive `await captureFailingCase(...)` inside the same `try` block that surrounds `pollResult`, an unexpected throw (e.g. a permissions error on `tests/`) would be caught by the existing `reportError`/`SolutionError` handling in `withFocusedSolution` and render as a Submit/Run *error*, even though the actual submission succeeded — directly violating ROADMAP success criterion 4.
**Why it happens:** The writer functions are designed never-throws (Pattern 1), but the *call site* in `submission.ts` must still not let a future violation of that contract propagate — defense in depth.
**How to avoid:** Keep the writer functions genuinely never-throws (no try/catch needed if they truly can't throw, matching `seedTests`/`addCase`), and additionally wrap the capture call itself in a narrow try/catch at the `submission.ts` call site that degrades to "no notes" on any unexpected exception, so a future regression in the writer can't take down run/submit.
**Warning signs:** A submit that clearly succeeded on LeetCode's side rendering as an error screen in leettui.

### Capture writer — submit-side new-case capture (D-01, D-03, D-04, D-05, D-06)
```typescript
// src/core/solutions/capture.ts — design sketch
import { existsSync, mkdirSync, readFileSync, writeFileSync } from "node:fs";
import { join } from "node:path";
import { addCase } from "./add-case";
import { compareOutput, discoverCases, normalize } from "../testRunner"; // normalize needs exporting

export type CaptureOutcome = "captured" | "blessed" | "duplicate" | "mismatch";
export interface CaptureResult { name: string; outcome: CaptureOutcome; note: string }

// input-only (D-03) when expectedOutput is undefined.
export function captureFailingCase(
  testsDir: string,
  input: string,
  expectedOutput?: string,
): CaptureResult {
  const target = normalize(input);
  const existing = discoverCases(testsDir).find(
    (c) => normalize(readFileSync(c.inputPath, "utf-8")) === target,
  );

  if (!existing) {
    const path = addCase(testsDir, input); // reuses 2.1's writer verbatim
    const name = path.split("/").pop()!.replace(".txt", "");
    if (expectedOutput !== undefined) {
      writeFileSync(join(testsDir, `${name}.out`), expectedOutput);
      return { name, outcome: "captured", note: `Captured ${name} (input + expected)` };
    }
    return { name, outcome: "captured", note: `Captured ${name} (input only)` };
  }

  if (!existing.expectedPath) {
    if (expectedOutput === undefined) {
      return { name: existing.name, outcome: "duplicate", note: `${existing.name} already captured` };
    }
    writeFileSync(join(testsDir, `${existing.name}.out`), expectedOutput); // D-05: fill, not clobber
    return { name: existing.name, outcome: "blessed", note: `Blessed ${existing.name}.out` };
  }

  if (expectedOutput !== undefined) {
    const prev = readFileSync(existing.expectedPath, "utf-8");
    if (!compareOutput(expectedOutput, prev)) {
      // D-06: never overwrite a disagreeing .out — surface, don't write.
      return {
        name: existing.name,
        outcome: "mismatch",
        note: `${existing.name}.out disagrees with LeetCode's expected output`,
      };
    }
  }
  return { name: existing.name, outcome: "duplicate", note: `${existing.name} already captured` };
}
```

### Run-blessing — content-matched, never index-matched (D-02, D-07)
```typescript
// src/core/solutions/capture.ts — design sketch (continued)
export function blessRunOutputs(
  testsDir: string,
  exampleTestcaseList: string[],
  expectedCodeAnswer: string[] | undefined,
): CaptureResult[] {
  if (!expectedCodeAnswer || expectedCodeAnswer.length === 0) return [];
  const cases = discoverCases(testsDir);
  const results: CaptureResult[] = [];
  for (let i = 0; i < exampleTestcaseList.length; i++) {
    const answer = expectedCodeAnswer[i];
    if (answer === undefined) continue;
    const target = normalize(exampleTestcaseList[i]!);
    const match = cases.find((c) => normalize(readFileSync(c.inputPath, "utf-8")) === target);
    if (!match || match.expectedPath) continue; // D-07: content match; skip if already has a .out
    writeFileSync(join(testsDir, `${match.name}.out`), answer);
    results.push({ name: match.name, outcome: "blessed", note: `Blessed ${match.name}.out` });
  }
  return results;
}
```
