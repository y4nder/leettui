# Phase 2.2: Auto-Capture Failing Cases - Context

**Gathered:** 2026-07-11
**Status:** Ready for planning

<domain>
## Phase Boundary

When a `run`/`submit` comes back with a failing verdict, leettui **auto-captures**
the failing LeetCode input and expected output from the already-parsed
`CheckResponse` into the problem's `tests/` dir as `case-NN.txt` (+ sibling
`case-NN.out` when the expected output is available) — turning any online failure
into a permanent, reproducible **offline** regression case gradeable by
`leettui test` (closing the online→offline debug loop opened by Phase 2.1).

Scope expansion agreed during discussion (still within the "capture from
run/submit into offline regression" boundary): the **run** path additionally
**blesses missing `.out` files for existing seeded cases** from the response's
`expected_code_answer[]` — LeetCode's authoritative expected outputs for the
example cases that today ship inputs-only.

**Not in this phase:** any new TUI surface beyond result-view text (no new
panels/modals), no changes to the offline runner itself, no background/retry
machinery. Capture is a side effect of the run/submit the user already invoked.

</domain>

<decisions>
## Implementation Decisions

### Capture triggers (which verdicts write files)
- **D-01:** **Submit wrong-answer** is the headline capture: `last_testcase` →
  new `case-NN.txt`, `expected_output` → sibling `case-NN.out` (subject to the
  dedup rules below). **The run path also participates**: run responses bless
  missing `.out`s for existing seeded cases from `expected_code_answer[]` —
  authoritative data, better than `--save`'s trust-the-user model.
- **D-02:** Run-blessing fires on **any completed run** (accepted *or*
  wrong-answer) — `expected_code_answer[]` is LeetCode-authoritative regardless
  of the user's verdict; an accepted run is in fact the safest blessing moment.
- **D-03:** **RE/TLE/MLE submits capture input-only**: `last_testcase` (present
  without `expected_output` on these verdicts) is written as `case-NN.txt` with
  **no** `.out` — offline `leettui test` immediately reproduces the crash/timeout
  (status `error`/`timeout`), and the `.out` is blessed later by a run or
  `--save` once fixed. Compile-error / pending / timeout / unknown responses
  carry no testcase and never capture.

### Dedup & conflict rules (non-destructive invariant)
- **D-04:** "Already captured" = **normalized input match** against every
  existing `case-NN.txt`: trailing-whitespace/newline + CRLF-insensitive
  comparison (same spirit as `compareOutput`'s lenient string fallback). Catches
  cosmetic-whitespace dupes without false positives.
- **D-05:** If the failing input matches an existing case that has **no**
  sibling `.out` and the response carries `expected_output`, **fill the missing
  `.out`** for that existing case instead of writing a duplicate — a fill, not a
  clobber; the case becomes gradeable.
- **D-06:** If the matching case already **has** a `.out` that **disagrees**
  with LeetCode's expected output, **never overwrite it** — surface a mismatch
  note (e.g. "case-03.out disagrees with LeetCode's expected output") so the
  user re-blesses deliberately via `test --save` or hand-edit. The auto path is
  strictly additive (ROADMAP success criterion 2).
- **D-07:** Run-blessing maps `expected_code_answer[i]` to the local case whose
  `.txt` content matches `exampleTestcaseList[i]` (**match by input content**,
  same normalization as D-04) — never by index/position, since the local
  `tests/` dir may hold user-added (`--add-case`) or 2.2-captured cases that
  weren't part of the run. An unmatched answer is simply not written.

### Hook placement & surfaces
- **D-08:** Capture hooks **inside the core submission service** —
  `runSolution`/`submitSolution` in `src/core/submission.ts` invoke the capture
  writer (which lives in `src/core/solutions/` beside `addCase`/
  `saveGoldenOutputs`) and return capture info alongside the `ParsedResponse`.
  The TUI (`R`/`s`) and the headless CLI (`leettui run`/`submit`) both get
  capture for free and can never drift — consistent with the service already
  doing side effects (`appendSubmissionRecord`, `markAttempted`).

### Defaults adopted without explicit confirmation
*(User went AFK for the last two questions; the recommended options were
adopted. Override by editing this file before `/gsd-plan-phase 2.2` if wrong.)*
- **D-09 (adopted by default):** **Always-on, no config knob.** Capture is
  additive-only, never overwrites, and degrades silently — nothing harmful to
  opt out of; matches leettui's lean/YAGNI philosophy. A `[test]` toggle can be
  added later if someone asks.
- **D-10 (adopted by default):** **Capture feedback = line(s) in the existing
  result view** (TUI fullscreen takeover / Result panel, and the CLI's
  `presentResultView`): e.g. "Captured tests/case-05 (input + expected)",
  "Blessed 2 expected outputs", and the D-06 mismatch note. Reuses the existing
  view-builder seam (`views/browse/resultView/builders.ts`, `cli/present.ts`);
  no new UI surface.

### Claude's Discretion
- Exact capture-module file naming/split inside `src/core/solutions/` and the
  shape used to thread capture results back to callers (extend the service
  return vs. a small wrapper type) — must keep `ParsedResponse` consumers
  compiling.
- Defensive handling when expected fields are unexpectedly absent (e.g. a
  submit-WA missing `expected_output` → degrade to D-03-style input-only
  capture; a missing `last_testcase` → no capture, result still renders).
- Exact message wording/glyphs for capture feedback — reuse `present.ts` /
  result-view semantics (`✓`/`!`/`·`, TTY-gated color).
- Whether run-blessing reports per-case outcomes or a single summary count.
- Test strategy details (pure writer unit tests + a mocked-response service
  test), matching the co-located `*.test.ts` convention.

<code_context>
## Existing Code Insights

### Reusable Assets
- `nextCaseName`/`addCase` (Phase 2.1) — sequential case numbering + input
  writing, already never-throws/create-if-absent; capture reuses rather than
  reimplements.
- `saveGoldenOutputs`'s `SaveOutcome` model — created/changed/unchanged/skipped
  per-case reporting maps directly onto capture feedback (captured / blessed /
  skipped-duplicate / mismatch).
- `compareOutput`'s normalization — the template for D-04's input matching
  (trailing-whitespace/CRLF-insensitive; JSON canonicalization on the `.out`
  side).
- `parseCheckResponse` — the verdict discrimination is already done; capture
  branches on `ParsedResponse.type`, no new response parsing.

### Established Patterns
- **Never-throws / best-effort / additive-only** core writers (`seedTests`,
  `addCase`, `saveGoldenOutputs`, `git.ts`) — capture must never destabilize
  run/submit (ROADMAP criterion 4): any parse/write failure degrades to "no
  capture", the result still renders.
- **Service-layer side effects** — `submitSolution` already appends DB records
  and marks status; capture is one more side effect at the same layer (D-08).
- **Both surfaces through one service** — TUI handlers and CLI verbs share
  `runSolution`/`submitSolution`, so one hook covers `R`/`s` and
  `:!leettui run`/`submit`.

### Integration Points
- New capture writer(s) in `src/core/solutions/` (beside `add-case.ts`/
  `save-output.ts`), co-located `*.test.ts`.
- Hook calls + capture-info return in `src/core/submission.ts`.
- Feedback lines in `views/browse/resultView/builders.ts` (TUI) and
  `cli/present.ts` (CLI) per D-10.
- Captured files land in the existing `getTestsDir(id, slug)` layout and are
  picked up by `discoverCases` with zero runner changes.

</code_context>
