**Never-throws / dir-injected / outcome-type pattern** (`save-output.ts` lines 33-49, mirrored by `add-case.ts` lines 31-36):
```typescript
export type SaveOutcome = "created" | "changed" | "unchanged" | "skipped";
export interface SaveResult { name: string; outcome: SaveOutcome; }

export function saveGoldenOutputs(testsDir: string, cases: LocalCaseResult[]): SaveResult[] {
  return cases.map((c): SaveResult => {
    if (c.status === "error" || c.status === "timeout" || c.actual === undefined) {
      return { name: c.name, outcome: "skipped" };
    }
    mkdirSync(testsDir, { recursive: true });
    const path = join(testsDir, `${c.name}.out`);
    const existed = existsSync(path);
    const prev = existed ? readFileSync(path, "utf-8") : null;
    writeFileSync(path, c.actual);
    if (!existed) return { name: c.name, outcome: "created" };
    return { name: c.name, outcome: compareOutput(c.actual, prev as string) ? "unchanged" : "changed" };
  });
}
```
`capture.ts` follows this exact shape: `(testsDir: string, ...) => CaptureResult[]`, no try/catch (synchronous `node:fs`, resolvable paths don't throw in practice — same reasoning documented in `save-output.ts`'s header comment), returns an outcome union rather than throwing. Extend the outcome vocabulary with a `mismatch` arm for D-06 (RESEARCH.md's design sketch, lines 296-372, already matches this shape almost verbatim — use it as the literal starting point).

### Content-based matching, never index-based
**Source:** `src/core/testRunner.ts` — `normalize` (line 87-89, to be exported), `discoverCases`/`pairCases` (lines 66-142), `compareOutput` (lines 120-128)
**Apply to:** `capture.ts`'s D-04/D-05/D-06/D-07 dedup/match logic — always compare via `normalize(readFileSync(...))`, never via array index against `case-(i+1)`.

### Service-layer side effect after `pollResult` resolves
**Source:** `src/core/submission.ts` — `appendSubmissionRecord` invocation inside `submitSolution` (line 117), `markAccepted`/`markAttempted` (lines 118-127)
**Apply to:** the new capture-writer invocation inside both `runSolution` and `submitSolution`, at the same point, wrapped in its own narrow try/catch (Pitfall 5) so a hypothetical writer regression can't turn a successful submit into an error screen.

### One `ResultView` field reaches every render surface
**Source:** `src/views/browse/resultView/types.ts` (`ResultView`'s existing optional fields), consumed identically by `src/ui/components/ResultBody.tsx` and `src/cli/present.ts`
**Apply to:** the new `notes?: string[]` field — one type addition, two renderer additions (dim/subtle styling in both), zero changes needed to `ResultPopup.tsx`/`ResultFullscreen.tsx`/`ProblemView.tsx` since they all consume `ResultBody` as a single shared component.

### Dual-surface call-site parity (TUI + CLI)
**Source:** `src/views/problem/handlers.ts` (`handleProblemRun`/`handleProblemSubmit`) and `src/cli/index.ts` (`runApiVerb`'s two `action` callbacks) both calling `runSolution`/`submitSolution` from `src/core/submission.ts`
**Apply to:** any return-shape change to `runSolution`/`submitSolution` — grep-verify (`grep -rn "runSolution(\|submitSolution(" src`) before editing to catch all call sites in the same commit (RESEARCH.md Pitfall 2 — confirmed exactly 2 production call sites plus their own tests).
