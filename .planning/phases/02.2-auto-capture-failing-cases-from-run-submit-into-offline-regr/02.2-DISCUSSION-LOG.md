## Capture trigger scope

**Q1: Should the run path also capture (expected_code_answer[] can bless seeded .out files)?**

| Option | Description | Selected |
|--------|-------------|----------|
| Submit-WA + run blessing (Recommended) | Submit-WA writes the new failing case; run responses additionally bless .out for existing seeded cases from expected_code_answer[] | ✓ |
| Submit-WA only | Strictly the roadmap's literal scope; run responses ignored | |
| Run-WA + submit-WA, no accepted-path blessing | Both WA verdicts capture, but an accepted run never writes | |

**Q2: When should run-blessing of seeded .out files happen?**

| Option | Description | Selected |
|--------|-------------|----------|
| Any completed run (Recommended) | Accepted or WA — expected_code_answer[] is authoritative regardless of verdict | ✓ |
| Only wrong-answer runs | Bless only on failure; happy path stays test --save | |

**Q3: Should RE/TLE/MLE submits (last_testcase, no expected_output) capture too?**

| Option | Description | Selected |
|--------|-------------|----------|
| Yes, input-only capture (Recommended) | Write case-NN.txt with no .out; offline test reproduces the crash/timeout | ✓ |
| Wrong-answer only | Drop RE/TLE inputs | |
| Capture RE only, skip TLE/MLE | RE reproduces cleanly; TLE/MLE depend on LeetCode's limits | |

## Dedup & conflict rules

**Q1: How do we decide a failing input is 'already captured'?**

| Option | Description | Selected |
|--------|-------------|----------|
| Normalized input match (Recommended) | Trailing-whitespace/newline + CRLF normalization against every existing case-NN.txt | ✓ |
| Exact byte match | Byte-for-byte only; a trailing newline diff would duplicate | |
| You decide | Planner discretion within the de-dup criterion | |

**Q2: Failing input matches an existing case with NO .out — what happens?**

| Option | Description | Selected |
|--------|-------------|----------|
| Fill the missing .out (Recommended) | Write the existing case's missing .out from expected_output — a fill, not a clobber | ✓ |
| Skip entirely | Write nothing even though the answer was in hand | |

**Q3: Existing .out disagrees with LeetCode's expected_output — resolve how?**

| Option | Description | Selected |
|--------|-------------|----------|
| Never overwrite, report mismatch (Recommended) | Auto path stays additive-only; user re-blesses deliberately | ✓ |
| Overwrite with LeetCode's answer | Authoritative but violates the never-overwrite criterion | |
| Overwrite only run-blessed .outs | Needs provenance metadata — too much machinery | |

**Q4: How to map expected_code_answer[i] to the right local case file?**

| Option | Description | Selected |
|--------|-------------|----------|
| Match by input content (Recommended) | Pair by normalized content match with exampleTestcaseList[i]; unmatched answers not written | ✓ |
| Match by index/position | Breaks silently once local cases diverge from the seeded set | |
