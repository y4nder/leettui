<objective>
Build the pure, dir-injected, never-throws capture engine that turns a failing (or
authoritative) LeetCode `CheckResponse` into local `tests/case-NN.txt` (+ `case-NN.out`)
files, and formalize the phase's requirements. This is the core of Phase 2.2's vertical
slice: `capture.ts` proves — via its own round-trip test — that a captured case is
immediately gradeable offline by `leettui test`. Plan 02 wires this engine into the
run/submit service and surfaces its feedback.

Purpose: Reuse Phase 2.1's primitives (`addCase`, `discoverCases`, `compareOutput`) under
the D-01..D-07 branching rules so no new "capture engine" abstraction is invented — just
two decision functions beside `add-case.ts`/`save-output.ts`.
Output: `src/core/solutions/capture.ts` + `capture.test.ts`, an exported `normalize`,
a barrel export, and TCASE-04/TCASE-05 formalized in REQUIREMENTS.md.
</objective>

## Phase Goal

**As a** LeetCode solver working from the terminal, **I want to** have leettui automatically
save a failing run/submit's input and expected output as an offline test case, **so that** I
can reproduce and fix the bug locally with `leettui test` without repeatedly hitting LeetCode.

> Note: the ROADMAP `**Goal:**` line for Phase 2.2 is outcome-shaped rather than in strict
> "As a / I want to / so that" form. The story above is derived faithfully from that goal and
> CONTEXT.md (user = terminal solver; capability = auto-capture failing cases offline; outcome =
> closed online→offline debug loop). Run `/gsd mvp-phase 2.2` if you want the ROADMAP goal line
> itself reformatted — no scope change implied.

| T-02.2-02 | Tampering | existing `.out` integrity | medium | mitigate | D-06: a matching case whose `.out` disagrees with LeetCode's expected output is never overwritten — a "mismatch" note is returned and the file is left byte-unchanged (asserted in `capture.test.ts`). |
