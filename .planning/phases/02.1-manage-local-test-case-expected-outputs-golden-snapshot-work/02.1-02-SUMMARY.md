# Phase 2.1 Plan 02: `leettui test --add-case` Summary

**`leettui test --add-case` reads a new case's input from stdin or a file argument and writes it as the next sequential `case-NN.txt` via a new dir-injected `addCase`/`nextCaseName` writer, closing the loop with plan 01's `--save` so a brand-new local test case can be added and blessed without hand-creating any files.**

## Accomplishments

- New core writer `nextCaseName(testsDir)`/`addCase(testsDir, input)` in `src/core/solutions/add-case.ts`, mirroring `seedTests`'/`saveGoldenOutputs`' never-throws, dir-injected idiom. `nextCaseName` computes `max(existing case numbers) + 1`, 2-digit-padded to match `seedTests`, and is gap-safe (a hole like `case-01`, `case-03` yields `case-04`, never gap-fill) — a Tampering mitigation (T-02.1-04): the written filename is never derived from user input.
- `addCase` writes ONLY the `.txt` input (never a `.out`, D-05) — the two-step flow is complete: `--add-case` creates the input, `leettui test --save` (plan 01) blesses its expected output.
- `leettui test --add-case` is now live in `src/cli/index.ts`: an early branch inside the `test` case (before `runLocalTests` runs) reads stdin (`Bun.stdin.text()`) or a file argument (`flagValue(argv, "--add-case")` + `Bun.file(path).text()`), rejects empty/whitespace input with the branded error + exit 2 (D-10, never creates a blank case), and otherwise writes the case and prints `created: <path>` with exit 0.
- Barrel re-export of `addCase`/`nextCaseName` from `core/solutions`; `src/cli/index.test.ts` pins the `--add-case` file-arg `flagValue` contract (file path resolves, bare form is `undefined`).
- TCASE-02 marked complete in `.planning/REQUIREMENTS.md` (all three TCASE requirements — TCASE-01/02/03 — are now complete).

## Decisions Made

- `addCase`/`nextCaseName` take an explicit `testsDir: string` (not `(id, titleSlug)`), matching `discoverCases`'/`saveGoldenOutputs`' dir-injected shape — unit-testable in-process with a temp dir; the CLI resolves `getTestsDir(id, slug)` and passes it in.
- `nextCaseName` is strictly `max + 1`, never gap-fill, so the numbering stays monotonic and collision-free even after a case is manually deleted, matching the plan's explicit behavior spec.
- `--add-case` is handled as the first branch inside the existing `test` case (before `runLocalTests`), since it's an alternate mode of `test` that writes an input and exits rather than running the solution — mirrors how `--save` is also a branch of the same case rather than a separate verb.
