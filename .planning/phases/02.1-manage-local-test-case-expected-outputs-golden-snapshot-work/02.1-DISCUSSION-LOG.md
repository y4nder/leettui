## Surface

| Option | Description | Selected |
|--------|-------------|----------|
| CLI verb only | Headless verb/flag run from the editor inner-loop; smallest surface, reuses runLocalTests + cwd-inference. | ✓ |
| TUI action only | A ProblemView key to accept the shown output; no editor round-trip but requires being in the TUI + a modal for add-case. | |
| Both CLI + TUI | Ship both over a shared core helper; most complete but ~double the work for a QoL phase. | |

**User's choice:** CLI verb only — and after clarifying what option 1 offers, specifically endorsed the `leettui test --save` (flag-on-`test`) shape over a separate `accept` verb ("the --save does feel right").
**Notes:** User first asked "what does option 1 offer?" before deciding; walked through the golden-snapshot loop (run `test` → eyeball → `test --save` → subsequent runs grade pass/fail). Resolves the ROADMAP's deliberately-open CLI-vs-TUI design note in favor of CLI-only; TUI action deferred.

## Accept-output semantics

| Option | Description | Selected |
|--------|-------------|----------|
| Only clean-run cases, skip errors/timeouts | Save `.out` for `ran`/`pass` cases; skip `error`/`timeout`, report skips. Never bless a crash. | ✓ |
| All cases unconditionally | Write `.out` for every case incl. error/timeout; simpler but can enshrine garbage as expected. | |
| Only cases missing a .out (never overwrite) | Fill only where absent; safe but can't update a golden after intentional behavior change. | |

| Option (overwrite) | Description | Selected |
|--------|-------------|----------|
| Overwrite (golden update), report changes | jest -u model: overwrite existing `.out`, print created/changed/unchanged/skipped. | ✓ |
| Overwrite only with a confirm/extra flag | Overwriting requires `--force`/y-N; extra guard at the cost of friction in the re-bless flow. | |

**User's choice:** Skip error/timeout cases; overwrite existing `.out` (golden-update) with per-case reporting.
**Notes:** Two sub-questions in this area (what-to-save, then overwrite behavior). Never-silent reporting is a firm requirement.

## Add-new-case flow

| Option | Description | Selected |
|--------|-------------|----------|
| Dedicated add flag reading stdin/file | `leettui test --add-case` reads input from stdin/file, writes next `case-NN.txt`; `--save` blesses output after. Two-step. | ✓ |
| Editor creates input, `--save` handles output | No add command; user creates `case-NN.txt` in the open workspace, `--save` fills `.out`. Leanest but drops "no hand-creating files". | |
| Add flag that also captures output in one shot | `--add-case` writes input AND immediately runs+writes `.out`. Most complete UX; most logic. | |

**User's choice:** Dedicated add flag reading stdin/file (two-step: add input, then `--save`).
**Notes:** Surfaced the tension that the lean `--save` almost makes a separate add command optional, but user chose the explicit flag to honor "without hand-creating files." One-shot add+capture noted as a deferred fallback.

---

## Correctness guard

| Option | Description | Selected |
|--------|-------------|----------|
| Trust-the-user + clear reminder line | No blocking guard; write goldens + print a reminder to verify correctness. Fits non-interactive editor loop + lean philosophy. | ✓ |
| Require an explicit confirm flag | `--save` previews; writing needs `--save --yes`. Deliberate second step, more friction. | |
| Warn only when overwriting a currently-failing golden | Write freely for missing `.out`; prominent warning when overwriting a disagreeing `.out`. | |

**User's choice:** Trust-the-user + a clear reminder line.
**Notes:** CLI runs non-interactively from `:!`, so a blocking prompt is awkward; correctness of the blessed output is explicitly the user's responsibility.
