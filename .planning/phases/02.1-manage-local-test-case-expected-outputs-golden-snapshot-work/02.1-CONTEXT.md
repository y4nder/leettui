# Phase 2.1: Local Test-Case Output Management - Context

**Gathered:** 2026-07-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Give the user a way to **populate and edit a case's expected output** (`case-NN.out`)
without hand-writing files, so the offline `leettui test` runner grades real
**pass/fail** instead of the verdict-less `ran`. The gap being closed: seeded
`tests/case-NN.txt` files ship **inputs only** (LeetCode's `exampleTestcaseList`
has no expected outputs), and `testRunner.ts` only diffs against an *optional*
sibling `.out`, so today every seeded case runs as `ran` and can never show green.

The surface for this phase is **CLI-only** ‚Äî a golden-snapshot workflow bolted
onto the existing headless `leettui test` verb, run from the editor inner-loop
(`:!leettui ‚Ä¶`). Two capabilities: (1) **snapshot** the current run's stdout into
`case-NN.out` for the cases that ran cleanly (`leettui test --save`), and (2)
**add a brand-new custom case** input (`leettui test --add-case`), whose output is
then blessed by a follow-up `--save`.

**Not in this phase:** any TUI/ProblemView surface (deferred ‚Äî see Deferred Ideas),
and the auto-capture of wrong-answer run/submit cases (that is Phase 2.2, which
reuses this phase's `.out`-writing seam).
</domain>

<decisions>
## Implementation Decisions

### Surface
- **D-01:** Surface is **CLI-only**, shaped as a **flag on the existing `test`
  verb** (`leettui test --save`), NOT a separate `accept`/`case` verb. One
  invocation runs the cases *and* snapshots their output in a single shot, and it
  slots into the `:!leettui test` inner-loop the user already runs from `$EDITOR`.
  Chosen over a TUI action (deferred) and over "both" (double the surface for a
  QoL phase). Reuses `runLocalTests` (already returns each case's actual stdout),
  `resolveTarget(cwd)` (problem/language inference), and the `tests/` write seam.

### `--save` semantics (which cases get a `.out`)
- **D-02:** `--save` writes `.out` **only for cases that ran cleanly** (status
  `ran` or `pass`). Cases that `error` or `timeout` are **skipped** ‚Äî a crash's
  stderr / partial output must never be blessed as the expected answer ‚Äî and the
  skip is **reported** in the output.
- **D-03:** For a clean-run case that **already has a `.out`**, `--save`
  **overwrites** it with the current output (the golden-update / `jest -u` model)
  ‚Äî re-blessing after an intentional behavior change is a first-class use case,
  not an error. **Every run reports** the per-case outcome (created / changed /
  unchanged / skipped) so a write is never silent.
- **D-04:** The **safety invariant** (from ROADMAP success criterion 3 + the
  never-throws core contract): `--save` **never touches an unrelated case** ‚Äî only
  cases discovered for the current run's `tests/` dir are written, and the writer
  is idempotent and best-effort (never throws). "Overwrite" in D-03 means the
  *same* case's own `.out`, not a sibling.

### Add-new-case flow
- **D-05:** A **dedicated add flag** (working name `leettui test --add-case`)
  reads the new case **input** from **stdin or a file argument**, writes it as the
  **next sequential** `case-NN.txt` in the problem's `tests/` dir, and stops there.
  This is a deliberate **two-step**: `--add-case` creates the input, then a normal
  `leettui test --save` runs the solution and blesses the new case's `.out`.
  Fully satisfies "add a case without hand-creating files."
- **D-06:** The new input file must be picked up by the existing
  `discoverCases`/`pairCases` on the next run with no runner changes ‚Äî i.e. it must
  match the `^case-\d+\.txt$` convention and slot into the existing lexical sort
  (numbering/zero-padding detail is Claude's discretion, D-09).

### Correctness guard
- **D-07:** **Trust-the-user, no blocking guard.** `--save` writes the goldens and
  prints a **plain reminder line** (e.g. "Blessed N outputs from the current
  solution ‚Äî make sure it's actually correct, e.g. accepted on LeetCode."). No
  `--yes`/confirm gate and no interactive prompt ‚Äî the verb runs non-interactively
  from `:!` inside an editor, and this matches leettui's lean, never-nag
  philosophy. Correctness of the blessed output is explicitly the user's call.

### Claude's Discretion
- **D-08:** Exact **flag/subcommand naming** (`--save` vs `--accept`; `--add-case`
  vs `--add` vs a `case add` form) ‚Äî the *shape* (flag on `test`, CLI-only) is
  locked; the exact spelling is planner's call, consistent with existing CLI
  conventions in `src/cli/`.
- **D-09:** **Case numbering** for a newly added case (next index, zero-pad width
  matching `seedTests`' `case-NN` output, collision handling) ‚Äî planner's call;
  must keep `discoverCases` ordering correct.
- **D-10:** Exact **exit codes** for the new paths within the established 0/1/2
  family (e.g. `--save` with nothing to write, `--add-case` reading empty stdin) ‚Äî
  reuse the `exitCodeForLocalRun` conventions; couldn't-do-anything ‚Üí 2, ran/wrote
  fine ‚Üí 0. Whether `--save` also *reports* per-case pass/fail after writing (it
  has just run them) is a UX detail left to the planner.
- **D-11:** Exact **reporting format** of the created/changed/unchanged/skipped
  summary and the reminder line ‚Äî reuse `cli/present.ts` glyphs/color semantics
  (`‚úì`/`‚úó`/`!`/`¬∑`, TTY-gated color).
- **D-12:** How `--save` composes with a **compiled language's one-shot build**
  (rust/csharp/java compile once before the case loop) ‚Äî `runLocalTests` already
  handles this; `--save` should consume its `LocalCaseResult[]` rather than
  re-running, so no new build logic is expected.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `runLocalTests` ‚Üí `LocalRunReport.ran.cases[]` ‚Äî already returns each case's
  actual stdout and status; `--save` filters these to `ran`/`pass` and writes
  their `actual` to `case-NN.out`. No re-run, no re-compile.
- `getTestsDir(id, slug)` + `seedTests`' create-if-absent/never-throws idiom ‚Äî the
  template for the new `.out`-writer and `--add-case` input-writer.
- `discoverCases`/`pairCases` ‚Äî already discover `case-NN` inputs and pair them
  with optional `.out`s; a written `.out` (or added `.txt`) is picked up next run
  with zero runner changes.
- `resolveTarget`/`resolveProblemTarget` ‚Äî cwd ‚Üí problem/language inference, no new
  targeting logic needed.
- `present.ts` ‚Äî shared CLI renderer (glyphs, TTY-gated color, exit codes).

### Established Patterns
- **Never-throws / best-effort / create-if-absent** core services (git.ts,
  harness runner, `deleteSolution`, `seedTests`) ‚Äî the new write helpers follow
  this: idempotent, never clobber an unrelated case, never crash the verb.
- **CLI exit-code family** 0 (ok) / 1 (ran-but-failed) / 2 (couldn't-run) ‚Äî new
  paths map into it (D-10).
- **Flag-on-existing-verb** rather than a new top-level verb ‚Äî keeps the
  `matchCliVerb` seam and editor muscle-memory (`leettui test ‚Ä¶`) intact.

### Integration Points
- New flag parsing in `src/cli/index.ts`'s `test` branch (`--save`, `--add-case`).
- New pure write helper(s) in `src/core/solutions/` (beside `seedTests`) ‚Äî
  co-located `*.test.ts`, testable without spawning.
- `--save` consumes `runLocalTests`' report; `--add-case` writes an input file
  then exits (output blessed by a later `--save`).
- Phase 2.2 will reuse the `.out`-write seam introduced here to auto-capture
  wrong-answer cases ‚Äî keep the writer a clean, reusable core function.

</code_context>

<specifics>
## Specific Ideas

- Golden-snapshot mental model is explicitly **`jest -u`**: `leettui test` shows
  output; `leettui test --save` blesses it; subsequent `leettui test` grades
  green/red and catches regressions offline.
- Command shapes discussed: `leettui test --save` (bless outputs),
  `leettui test --add-case` (add a new input from stdin/file). Exact spelling is
  D-08 (discretion).
- Reminder-line intent (D-07): something like "Blessed N outputs from the current
  solution ‚Äî verify it's actually correct (e.g. accepted on LeetCode)."

</specifics>

<deferred>
## Deferred Ideas

- **TUI ProblemView "accept output as expected" action** ‚Äî a key on the `t`
  local-test result to bless output / add a case from inside the TUI, over a
  shared core write-helper. Deliberately deferred to keep this QoL phase CLI-only;
  the core `.out`-writer built here makes a later TUI re-skin cheap (same pattern
  the recents modal/data-layer split used in Phase 2).
- **One-shot add+capture** (`--add-case` that also immediately runs the solution
  on just the new case and writes its `.out` in one command) ‚Äî considered but the
  simpler two-step (add input, then `--save`) was chosen; revisit if the two-step
  proves clunky.

Discussion stayed within the Phase 2.1 case-output-management boundary ‚Äî no scope
creep into Phase 2.2 (auto-capture) or a dashboard.

</deferred>
