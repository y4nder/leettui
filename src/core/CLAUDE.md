# core/

Business logic that bridges the API, database, and UI layers.

## Files

- `sync.ts` — `syncQuestions(onProgress?)` fetches all LeetCode problems in 1000-item pages, upserts into SQLite within a transaction. `syncIfEmpty()` only syncs if DB has zero questions (first run).
- `search.ts` — `filterQuestions(questions, needle)` performs fuzzy matching against question title and ID. Returns scored and sorted results. Substring matches score highest.
- `solutions/` — Solution file manager for the `~/.local/share/leettui/solutions/` directory. A **barrel module** (`index.ts` re-exports the public surface unchanged, so importers still use `from "../core/solutions"`), split by lifecycle into `paths.ts` (path math + `resolveProblem*`), `discovery.ts` (read/exists/list), `create.ts` (file/harness/tests/notes creation + the `createSolutionWithHarness` orchestrator), `remove.ts` (`deleteSolution` — lang-folder removal), and `templates.ts` (`renderTemplate`/`overlayTemplates`). **Layout (Stage 7):** `{paddedId}_{slug}/{langSlug}/solution.{ext}`, with shared `notes.md` + `tests/` reserved at the problem-folder level. The **langSlug is a solution's identity** (the directory name) — the filename is always `solution.{ext}`, so callers pass langSlugs, never filenames.
  - `getProblemDir(id, slug)` — `solutions/{paddedId}_{slug}` (no mkdir)
  - `resolveProblemPath(path, solutionsDir)` / `resolveProblemFromCwd(cwd)` (Stage 8) — the **reverse** of `getProblemDir`: parse a filesystem path back into `{ questionId, titleSlug, langSlug }` by matching the `{solutionsDir}/{paddedId}_{slug}/{langSlug}/…` layout. Pure + config-free (`resolveProblemPath` takes the root explicitly, so it's unit-tested with a fabricated dir in `solutions.test.ts`); the `…FromCwd` wrapper passes `getSolutionsDir()`. `langSlug` is `null` at the problem-folder level; returns `null` when the path is outside the solutions tree or its first segment isn't a problem dir. **The DB lookup (id/slug → `DbQuestion`) is intentionally left to the caller.** Backs the headless CLI's cwd-inference, paired with `e` now spawning `$EDITOR` with `cwd` = the lang folder (`views/{browse,problem}/handlers.ts`)
  - `getSolutionPath(id, slug, langSlug)` — `…/{langSlug}/solution.{ext}`; mkdirs the lang subfolder
  - `getNotesPath(id, slug)` / `getTestsDir(id, slug)` — shared problem-level locations
  - `readNotes(id, slug)` → notes.md content or `null` (no side effects); `ensureNotesFile(id, slug, title?)` — mkdir + create-if-absent (seeds a minimal `# {id}. {title}` header on first creation only), returns the path ready for `$EDITOR`. Back the per-problem notes view (Stage 7 item 2); creating `notes.md` never flips the "has solution" marker since `langSlugsWithSolution` only counts lang subdirs
  - `getProblemMdPath(id, slug)` → `{problemDir}/problem.md` (sibling to `getNotesPath`); `ensureProblemDir(id, slug)` — mkdir the problem-folder level so a workspace can be opened before any solution exists; `ensureProblemMd(id, slug, descriptionMarkdown, title?)` — mkdir + **create-if-absent** (writes a `# {id}. {title}` heading above the description on first creation only; an existing/edited `problem.md` is never touched); `readProblemMd(id, slug)` → content or `null`. **Layout (Stage 13):** `problem.md` (offline problem statement) and `notes.md` (shared scratch) both live at the problem-folder level beside the language subfolders — the "open workspace" (`w`) action puts all three in the editor's file tree at once. `problem.md` is **tied to workspace-open**, not the `e` create flow (which has no description fetch); like notes, creating it never flips the "has solution" marker (it's a problem-level file, not a lang subdir — asserted in `solutions.test.ts`)
  - `createSolutionFile(id, slug, langSlug, code)` — writes the file only if it doesn't exist
  - `createSolutionWithHarness(id, slug, langSlug, code, metaDataRaw?, exampleTestcases?)` — create-flow orchestrator used by both create call sites (browse `handleOpenEditor`, problem `handlePickerOpenEditor`). Order: (1) **overlay per-language template overrides** (Stage 7 item 6) from `~/.config/leettui/templates/{langSlug}/`, then fill gaps with bundled defaults — (2) the solution file (LeetCode snippet, unless a `solution.{ext}` template supplied one), (3) the generated harness **file set** — interpreted langs emit one file (`main.py`/`main.js`/`main.ts`), **rust** emits three (`Cargo.toml` + `.gitignore` + `main.rs`); the orchestrator **loops** `harness.files`, each create-if-absent + skipped if a same-named template already supplied it — (4) seed `tests/` from the example cases. Stays network-free; the handlers fetch `consolePanelConfig` (for `metaData` + `exampleTestcaseList`) and pass the data in
  - **Template overrides** (Stage 7 item 6) — `overlayTemplates(srcDir, destDir, vars)` renders every file in a per-language template dir into the lang folder (create-if-absent, returning the set of filenames it "owns") and `renderTemplate(text, {functionName?, titleSlug})` substitutes `{{functionName}}`/`{{titleSlug}}` (functionName from `metaData`, empty when absent). **Filename convention** (the user-facing contract): a `solution.{ext}` template overrides the LeetCode snippet, a harness-named template (`main.py`/`main.js`/`main.ts`) overrides the generated harness, and **any other filename** (e.g. `Cargo.toml`, `build.rs`, helper modules) lands purely additively. **Rust `solution.rs` template caveat (Stage 21):** a template `solution.rs` is written **un-shimmed** (templates win untouched — `prepareRustSolution` is only applied to the *default* snippet path), but the generated `main.rs` now does `mod solution; use solution::Solution;` and no longer defines `struct Solution;` itself. So a rust `solution.rs` template that is a *bare* `impl Solution` will fail to build (`E0412 cannot find type Solution`) — it must carry its own `#[cfg(feature = "harness")] pub struct Solution;` (or override `main.rs`/`Cargo.toml` too). This is the one create path the shim-coupling can't reach without breaking the "templates win untouched" invariant. **Top-level files only** — subdirectories in the template dir are not recursed (matches the flat lang-folder model). Overlay runs first; because every default writer is create-if-absent, the ordering is the real safety net and the "applied set" check is belt-and-suspenders. Tests: `solutions.test.ts` (pure `renderTemplate`/`overlayTemplates` in-process; the create-flow end state via a subprocess with a temp `$HOME`, since Bun fixes `os.homedir()` at launch). **`core/harness/index.ts` was deliberately not touched** — routing harness overrides through `generateHarness` would force fs lookups + substitution into its pure, never-throws contract; the unified overlay handles the harness as just another (specially-named) template file
  - `getLangFilePath`/`getHarnessPath`/`createHarnessFile(id, slug, langSlug, filename, content)` — write an arbitrary file inside the lang folder (`getHarnessPath` delegates to `getLangFilePath`); `createHarnessFile` is create-if-absent (never clobbers user edits)
  - `seedTests(id, slug, exampleTestcases)` — writes `tests/case-NN.txt` from LeetCode example cases (each entry written verbatim, create-if-absent; empty list is a no-op)
  - `readSolutionFile(id, slug, langSlug)` — reads solution code (no side effects)
  - `findExistingSolutions(id, slug)` — returns the **langSlugs** that have a `solution.*` file (skips the shared `tests/`/`notes.md`); both this and `listSolutionQuestionIds` share one private `langSlugsWithSolution(dir)` predicate so the marker can never disagree with the problem view
  - `listSolutionQuestionIds()` — returns the set of question IDs whose problem folder contains at least one `solution.*` file (backs the question-list "solution exists" marker); a folder holding only `notes.md`/`tests/` is not counted
  - `deleteSolution(id, slug, langSlug)` (Stage 16, `remove.ts`) — removes a **single language subfolder** (`{paddedId}_{slug}/{langSlug}/` — its `solution.{ext}`, generated harness/manifest, and compiled `target/`), leaving the shared problem-level `notes.md`/`problem.md`/`tests/` **and every other language** untouched. **Idempotent + never-throws** (a missing folder is a no-op via `rmSync` `force`/`recursive`), mirroring the `migration.ts`/`relocate.ts` best-effort contract. **Safety invariant:** an **empty `langSlug` is refused** (returns early) — it would make the target path equal the problem dir and the recursive remove would wipe the shared notes/tests, the exact thing the lang-folder granularity protects. Backs the in-TUI delete action (`d` from the focused Solutions panel + a confirm modal — `views/problem/handlers.ts`); deleting a problem's **last** language drops its id from `listSolutionQuestionIds`. Subprocess + temp-`$HOME` tested in `solutions.test.ts`
- `harness/` — Local test-harness generation (Stage 7 items 3 + 5 + TS; Stage 14 rust; Stage 17 csharp; Stage 25 java). `index.ts` exposes `generateHarness(langSlug, metaDataRaw)` → `GeneratedHarness | null`, where `GeneratedHarness` is **multi-file** (`{ files: { filename, content }[] }`): interpreted langs return a one-element `files` array, a compiled lang returns several (rust → `Cargo.toml` + `.gitignore` + `main.rs`; csharp → `main.csproj` + `.gitignore` + `main.cs`; java → `.gitignore` + `main.java`, manifest-free since `javac` compiles sources directly). It dispatches by language (`python3` → `python.ts`, `javascript` → `javascript.ts`, `typescript` → `typescript.ts`, `rust` → `rust.ts`, `csharp` → `csharp.ts`, `java` → `java.ts`) and **never throws** (returns `null` on an unsupported language, missing/unparseable `metaData`, an interpreted-language signature touching `ListNode`/`TreeNode` (refused via the shared `hasDeferredType` — no broken passthrough), or a rust/csharp/java signature whose types can't be mapped, so solution creation can't fail on it). `meta.ts` holds the language-agnostic `MetaData` type + `parseMetaData` + `baseType` + `DEFERRED_TYPES` (the `ListNode`/`TreeNode` set) + `hasDeferredType(meta)` (true when any param/return touches a `DEFERRED_TYPES` member — the interpreted langs' refusal gate, used by `index.ts`) + the `HarnessFile` (`{ filename, content }`) output shape — language-agnostic so both the dispatcher and every generator share it. `python.ts` (`generatePythonHarness`, unit-tested in `python.test.ts`) emits a `main.py` that reads one JSON value per line from **stdin**, `json.loads`-es each into the positional args, calls `Solution().{fn}(...)`, and prints `json.dumps(result)`. `ecma.ts` holds the shared JS/TS skeleton (`renderArgReads` + `generateEcmaHarness`): both read one JSON value per stdin line, call `{fn}(...)`, and print `JSON.stringify(result ?? null)`, differing only in their `loadSolution()`. `javascript.ts` (`main.js`, run by `node`) loads the function via `new Function(src + "return {fn};")` over the raw `solution.js` text — LeetCode JS snippets are bare un-exported functions and the file must stay submittable, so `require` is out. `typescript.ts` (`main.ts`, run by `bun`) does the same but first strips types with `new Bun.Transpiler({ loader: "ts" }).transformSync(src)` — raw type annotations are invalid JS and would throw `Unexpected token ':'` in `new Function`. Both keep `solution.{js,ts}` byte-for-byte submittable; JS + TS are unit + end-to-end tested (`javascript.test.ts`, `typescript.test.ts` — the TS e2e uses a genuinely *typed* solution so it actually exercises the transpile). An interpreted-language signature touching a `DEFERRED_TYPES` member (`ListNode`/`TreeNode`) is **refused** by the dispatcher (`hasDeferredType` → `null`, no harness file) rather than emitting a broken passthrough that would hand a raw `[1,2,3]` to a node-expecting fn and crash; the runner then surfaces the actionable no-harness message (names the language, points to `R`/`s`). Real per-language list/tree deserializers stay deferred — demand-driven, built when actually hit (see `docs/scratchpad/notes.md` "Harness type-support strategy"). `rust.ts` (Stage 14, the first **compiled, statically-typed** language) is two layers: a pure LeetCode→Rust **type map** (`rustType`/`rustReturnType`/`isRustMappable`/`renderRustArgReads` — scalars `integer→i32`/`long→i64`/`string→String`/`double→f64`/`boolean→bool`/`character→char`, `T[]`/`T[][]` → `Vec<…>`, `void`→`()`) and `generateRustHarness(meta)` → `HarnessFile[] | null` that assembles `Cargo.toml` (`[[bin]] name="main" path="main.rs"` + `serde_json`), `.gitignore` (`/target`), and `main.rs` (`mod solution; use solution::Solution;` + a `main()` that `serde_json::from_str`-deserializes each stdin line to the arg's **concrete** type, calls `Solution::{fn}` — **snake_cased**, since `metaData.name` is camelCase but LeetCode's rust impl is snake_case — and `serde_json::to_string`s the result). **Stage 21 (LSP fix):** `main.rs` pulls `solution.rs` in as a **real module** (`mod solution;`), not `include!`, because rust-analyzer gives **0 completions** in `include!` targets — completion needs the editing file to be a module in the crate tree, and an `include!` target is macro *input*, not a module (hover/goto worked, completion was dead; diagnosed empirically, see `docs/scratchpad/rust-harness-lsp.md`). To make that module self-contained, `solution.rs` is seeded via `prepareRustSolution(code)` (dispatched by `index.ts`'s `prepareSolutionSnippet(langSlug, code)`, rust-only) which prepends `#[cfg(feature = "harness")] pub struct Solution;`, and `Cargo.toml` declares a **default-on `harness` feature**. Locally the feature is on → the struct exists → `mod solution;` resolves → completion works (0 → 234 items, verified); on LeetCode the feature is undefined → the line compiles to nothing → LeetCode's own `struct Solution;` is used, so `solution.rs` stays **submittable verbatim**. A declared feature (not a raw `--cfg harness`) is warning-clean and rust-analyzer enables it automatically, so the runner is unchanged (plain `cargo build` enables it). The snippet transform is **coupled to harness presence** in `create.ts` (`harness ? prepareSolutionSnippet(...) : code`) so an unmappable signature — which has no manifest declaring the feature — gets the plain snippet, never an orphaned feature-gated line. Like the interpreted langs (which now refuse outright), a `ListNode`/`TreeNode` (`DEFERRED_TYPES`) in any param or the return makes the signature **unmappable** → `generateRustHarness` returns `null` → blank `solution.rs` stub (rust additionally rejects unknown scalars via `isRustMappable`), because a non-mappable type is a hard *compile* error in rust, not a runnable passthrough. Unit + cargo-gated e2e tested (`rust.test.ts`). `csharp.ts` (Stage 17, the **second** compiled language) mirrors `rust.ts`: a pure LeetCode→C# **type map** (`csharpType`/`csharpReturnType`/`isCsharpMappable`/`renderCsharpArgReads` — scalars `integer→int`/`long→long`/`string→string`/`double→double`/`boolean→bool`/`character→char`, `T[]`/`T[][]` → C# `T[]`/`T[][]`, `void` kept literal) and `generateCsharpHarness(meta)` → `HarnessFile[] | null` assembling `main.csproj` (`<OutputType>Exe</OutputType>` + an **SDK-adaptive** `TargetFramework` (`net$(NETCoreSdkVersion.Split('.')[0]).0` → `net9.0`/`net10.0`/… matching the installed SDK major, so the framework reference pack always resolves — a hardcoded `net9.0` produced CS0518 "System.Object is not defined" in the editor C# LSP on machines whose only SDK was a different major) + `<ImplicitUsings>enable</ImplicitUsings>` + `<AssemblyName>main</AssemblyName>` — **no `PackageReference`**, since `System.Text.Json` ships in the framework, so the first build is **offline**, unlike rust's serde fetch), `.gitignore` (`bin/`/`obj/`/`out/`), and `main.cs` (a `Program.Main()` that `JsonSerializer.Deserialize<T>`s each stdin line to the arg's **concrete** type, calls `new Solution().{Method}` — **PascalCased**, since `metaData.name` is camelCase but LeetCode's C# method is PascalCase — and `JsonSerializer.Serialize`s the result; a `void` return calls without assignment and prints `null`, the analogue of rust's `()`). `solution.cs` stays the bare, **byte-for-byte submittable** `public class Solution` — **co-compiled** by the project (`dotnet build` compiles every `.cs` in the dir into one assembly), so no `include!`/`new Function` wrapper is needed; the harness just references `new Solution()`. `ImplicitUsings` mirrors LeetCode's C# env so a bare snippet using `List<>`/`Dictionary<>` compiles. Like rust, a `ListNode`/`TreeNode` (`DEFERRED_TYPES`) anywhere in the signature makes it **unmappable** → `null` → blank `solution.cs` stub. Unit + dotnet-gated e2e tested (`csharp.test.ts`). `java.ts` (Stage 25, the **third** compiled language) mirrors `csharp.ts`'s shape — a pure LeetCode→Java **type map** (`javaType`/`javaReturnType`/`isJavaMappable`/`renderJavaArgReads` — scalars `integer→int`/`long→long`/`string→String`/`double→double`/`boolean→boolean`/`character→char`, `T[]`/`T[][]` → Java `T[]`/`T[][]`, `void` kept literal) and `generateJavaHarness(meta)` → `HarnessFile[] | null` — but diverges on the JDK's **lack of a JSON library** (csharp's `System.Text.Json` is in-framework; rust fetches serde). Instead of pulling in Maven/Gradle + Gson, `main.java` **embeds a dependency-free JSON parse/serialize** that is *generic + reflective*: per-arg codegen is one line (`{T} {name} = ({T}) convert(parseJson(lines[{i}]), {T}.class);` — Java class literals like `int[][].class` are valid, so the fixed `convert`/`toJson`/`quote`/`parseJson` block covers every mapped type with no per-type generation; `convert` uses `java.lang.reflect.Array` to build typed arrays and unboxes scalars via the cast, `toJson` reflectively serializes back). The harness class is a **package-private `class Main`** (Java forbids `public class Main` outside `Main.java`; the launcher only needs `main` public, not the class), **co-compiled** with `solution.java`'s package-private `class Solution` (`javac solution.java main.java`) and run as `java -cp . Main` (the explicit `-cp .` survives an exported `CLASSPATH` that would otherwise drop the implicit cwd entry). **No name-case conversion** — LeetCode's Java method is already camelCase, same as `metaData.name` (unlike rust's snake_case / csharp's PascalCase). `prepareSolutionSnippet("java", …)` prepends `import java.util.*;` to `solution.java` (reusing the rust-shim seam, coupled to harness presence in `create.ts`) — the Java analogue of csharp's `<ImplicitUsings>`, so a bare snippet using `HashMap`/`List` compiles under local `javac` (LeetCode auto-imports `java.util`); it stays **submittable** (a redundant import is legal Java) but, like rust's shim, the on-disk file is no longer byte-identical to the starter. Like rust/csharp, a `ListNode`/`TreeNode` makes the signature **unmappable** → `null` → blank `solution.java` stub. Unit + JDK-gated e2e tested (`java.test.ts` — the e2e drives a `HashMap`-based `solution.java` written through `prepareJavaSolution` so it genuinely exercises the import prepend). `index.ts` also exposes `getRunnerSpec(langSlug)` → `{ harnessFilename, command, compile? } | null` (the `RUNNERS` registry: `python3 → main.py`/`["python3"]`, `javascript → main.js`/`["node"]`, `typescript → main.ts`/`["bun"]`, `rust → main.rs`/`["./target/debug/main"]` **plus** `compile.command = ["cargo","build","--quiet"]`, `csharp → main.cs`/`["./out/main"]` **plus** `compile.command = ["dotnet","build","-o","out","--nologo","-v","q"]` — `out/main` is the apphost named by `<AssemblyName>`, and `java → main.java`/`["java","-cp",".","Main"]` **plus** `compile.command = ["javac","solution.java","main.java"]` — manifest-free, the `.class` files land in cwd and `Main` is the package-private launcher class) — the seam the runner consults, so a new language is one `case` + one entry and the runner stays unchanged. The optional **`compile`** phase (Stage 14) marks a *compiled* language: present → build once, then run the produced binary per case; absent → interpret the harness file per case (see `testRunner.ts`). Each `command`/`compile.command` is a toolchain the user needs on PATH — `python3`/`node`/`bun`/**`cargo`**/**`dotnet`**/**`javac`+`java`** (note `bun`, `cargo`, `dotnet`, and the JDK aren't put on PATH by the shipped `--compile` binary, the same caveat as `node`/`python3`); a missing **interpreter** degrades to a per-case `error`, a missing **compiler** to the one-shot `compile-error` report (the `toolchainMissing` view hint is per-language: rust → rustup, csharp → the .NET SDK, java → a JDK).
- `testRunner.ts` — Local test runner (Stage 7 item 4; Stage 14 compile phase). `runLocalTests(question, langSlug)` resolves `getRunnerSpec`, then per case spawns the runner with **cwd = the lang folder** (so the harness imports/links the sibling `solution.*`) and the `tests/case-NN.txt` file on **stdin**, draining stdout/stderr; a 10s `TIMEOUT_MS` kills runaway cases. The per-case argv is `[...command, harnessFilename]` for an **interpreted** language but just `[...command]` for a **compiled** one (`spec.compile` present) — there `command` already *is* the produced binary (`./target/debug/main`), which reads stdin directly, so appending the source file would be wrong; `harnessFilename` still anchors the `existsSync` no-harness precheck for both. **Compiled languages** (Stage 14 — rust; Stage 17 — csharp; Stage 25 — java) run `spec.compile.command` **once** before the case loop (after the no-cases check, so an empty `tests/` skips a pointless build) with its own generous `COMPILE_TIMEOUT_MS` (120s — the first `cargo`/`dotnet`/`javac` build can far exceed the per-case 10s); a non-zero build, a missing toolchain (spawn throws → `cargo` not on PATH), or a build timeout short-circuits to a single `compile-error` report instead of N identical per-case errors. Each case gets a `CaseStatus`: `pass`/`fail` (only when an optional sibling `case-NN.out` exists to diff against), `ran` (executed, nothing to compare), `error` (non-zero exit), `timeout`. Returns a `LocalRunReport` discriminated union (`unsupported`/`no-harness`/`no-cases`/`compile-error`/`ran`); the `compile-error` arm carries `langSlug` + `toolchainMissing` + `output` and is consumed by **both** `buildLocalRunView` (`views/browse/resultView/builders.ts` → a **per-language** install hint when the toolchain is missing — rust → rustup, csharp → the .NET SDK, java → a JDK — else the compiler diagnostics) and `cli/present.ts` (`exitCodeForLocalRun` → **2**, the "couldn't run" family). **Key UX rule:** seeded cases ship inputs only (LeetCode gives no expected outputs), so a run with no `.out` files renders as info ("Ran N cases (no expected outputs)"), never a green pass — `buildLocalRunView` in `views/browse/resultView/builders.ts` derives `kind`/title from whether any case had a verdict. Pure parts (`pairCases`, `compareOutput` — **JSON-normalized**: the harness emits compact `[0,1]` but a copied `.out` uses display spacing `[0, 1]`, so both sides are canonicalized (recursive key sort, float-normalizing) before comparing; falls back to trailing-whitespace/CRLF-insensitive string equality when either side isn't valid JSON) unit-tested in `testRunner.test.ts`.
- `migration.ts` — `migrateSolutionsLayout(dir?)`: one-time, **idempotent**, **collision-safe** migration of legacy flat files (`0001_two-sum.py`) into the folder layout. Skips unknown-extension/non-matching files (loss-free), never overwrites an existing target, and continues past per-file errors. Run once at boot in `BootFlow` after `openDatabase`. Takes an optional dir for testing (`migration.test.ts`).
- `relocate.ts` — `relocateSolutions(fromDir, toDir)` (Stage 10 item 1): moves an entire solutions tree between directories, reusing `migration.ts`'s **idempotent** / **collision-safe** / best-effort-never-throw patterns. The unit of move is each **top-level entry** (a whole `{paddedId}_{slug}` problem folder — which already carries its nested `notes.md`/`tests/`/lang dirs — plus any other entry like a `.git` repo the user keeps there), so a git-tracked solutions dir relocates with its history. `renameSync` per entry; on **EXDEV** (cross-filesystem — the headline "point solutions at an external repo/mount" case) it falls back to `cpSync` + `rmSync` (the `existsSync(dest)` collision pre-check guarantees a clean target first). No-ops on `from === to`, a missing source, or a **nested** pair (one path inside the other — refused to avoid a self-referential move). Returns `{ moved, skipped }` (top-level entries, not problem count). Pure/dir-injected → unit-tested in `relocate.test.ts`. Item 2 added two more pure helpers here: `countProblemFolders(dir)` (top-level `/^\d+_/` **directories**, regardless of contents — a notes-only folder counts, unlike `listSolutionQuestionIds`) and `detectSolutionsRelocation(currentDir, lastKnownDir?) → RelocationPlan | null` (a plan only when last-known is a real, *different*, still-existing dir with ≥1 problem folder; `count` is a **pre-skip upper bound** since the move skips target collisions). Boot wiring for item 2 lives in `BootFlow`. The in-TUI change action (item 4) is the `changeSolutionsDir(rawInput, deps?)` orchestrator in `views/browse/handlers.ts` (persist-then-relocate, reading the current dir **before** persist resets the config memo — unit-tested via injected deps in `changeSolutionsDir.test.ts`), surfaced by `ChangeLocationPrompt` on the `"relocate"` ui mode.
- `git.ts` — Git scaffolding + delegation seam (Stage 22). **Never-throws** like `testRunner`: every git/`gh` op shells out via `Bun.spawn` (capturing stdout/stderr), and a missing binary returns `{ ok:false, reason:"tool-missing" }` instead of throwing, so the UI shows an install hint. Pure / PATH helpers: `commandExists(bin)` (`Bun.which`), `ghAvailable()`, `ghAuthed()` (`gh auth status` — installed≠logged-in), `ghCreateArgv(name,{private})` (the `gh repo create … --source=. --remote=origin --push` argv, unit-tested **without** spawning since a live run would create real repos), `manualRemoteInstructions(name)` (the printed fallback), and `ROOT_GITIGNORE`/`writeRootGitignore(dir)` (the defensive secrets `.gitignore` — `*.db`/`session.json`/`config.toml` — create-if-absent, never clobbers). Local git: `isGitRepo(dir)`, `gitInit(dir)`, `initialCommit(dir,msg)` (**idempotent** — skips the commit when nothing is staged, so a re-run on a clean tree doesn't error), and the single **`ensureSolutionsRepo(dir)`** seam (mkdir → init-if-needed → defensive `.gitignore`-if-absent) that both the init prompt and the remote wizard call, so the gitignore-before-`add` ordering can't diverge (the security invariant — the wizard is the only path that pushes). `createGitHubRepo(dir,name,{private})` runs the gh argv. **Sync FROM a remote (Stage 24):** `run()` gained an optional `env` (merged over `process.env`) so the network ops set non-interactive git vars — `GIT_TERMINAL_PROMPT=0` (+ `GIT_SSH_COMMAND="ssh -o BatchMode=yes"`); git prompts on `/dev/tty` directly (NOT the `stdin:"ignore"`), so without this a private-repo credential/host prompt would hang the live TUI. The seam adds `isDirEmpty(dir)` (absent or zero entries — strict, counts dotfiles; the clone target must be empty, which doubles as the **data-safety guard** against clobbering a solutions dir repointed at the data dir), `hasRemote(dir)` (gates the pull path), `ghCloneArgv(repo,dir)` (pure, unit-tested like `ghCreateArgv` — no live run), `cloneGitHubRepo(repo,dir)` (gh's own auth + the non-interactive env), `gitPull(dir)` (`git pull --ff-only` — the safe fast-forward boundary; a diverged history fails so the merge delegates to lazygit), and `manualCloneInstructions(repo,dir)` (the printed gh-absent fallback). Consumed by `views/browse/handlers.ts` (`handleOpenGitUi` → suspend/spawn the git UI · `scaffoldSolutionsGit`/`handleConfirmGitInit` · `runRemoteSetup` · the Stage 24 `handleOpenGitSync`/`runRemoteSync`/`runRemotePull`). Unit + git-gated e2e in `git.test.ts` (the Stage 24 `hasRemote`/`gitPull` e2e use **local** repos — fast-forward success and the diverged refusal — no network/gh).
- `session.ts` — Persists small cross-run state to `~/.local/share/leettui/session.json`: the last-viewed browse position (`{ topicSlug, questionId }`, Stage 5) and the last resolved solutions dir (`solutionsDir`, Stage 10 — `getLastKnownSolutionsDir()` returns `undefined` when never recorded, so item 2 can tell "first boot" from "the path changed"; `setLastKnownSolutionsDir(dir)` is a synchronous merge-write). All writers **merge** into one lazily-disk-loaded in-memory mirror (`_state`), so a partial update never clobbers another feature's field; the debounced position write (`saveSession`, ~400ms because navigation fires per keypress) serializes the *live* mirror at fire time, not a stale snapshot, so a synchronous `setLastKnownSolutionsDir` landing mid-debounce survives. `getLastShownChangelogVersion()`/`setLastShownChangelogVersion(tag)` (Stage 18) mirror the solutions-dir accessors (synchronous merge-write) — the persistent per-version flag that gates the "What's new" popup to once per release. `loadSession()` returns the mirror. Merge invariants covered in `session.test.ts` (subprocess + temp `$HOME` for a real cross-boot, incl. the changelog-version field coexisting with position + solutions dir).

- `markdown.ts` — `htmlToMarkdown(html)`: a single shared, configured `node-html-markdown` converter for LeetCode descriptions. Maps `<sup>`/`<sub>` to `^`/`_` (so exponents like `10^5` survive), drops `<img>`, fences code blocks, normalizes `&nbsp;`. Used by both browse and problem handlers.
- `clipboard.ts` — `copyToClipboard(text)` via the OSC 52 terminal escape (works over SSH, no external helper); `problemUrl(slug)` builds the canonical problem URL.
- `update.ts` — Self-update + update notification. `runUpdate({force})` (the `update` subcommand) downloads the latest release asset and atomically swaps it over the running binary; gated on `IS_RELEASE`. **Stage 19 (compressed assets):** `assetUrls(tag, asset)` is the pure `{ gz, raw }` URL pair (unit-tested); `fetchAsset(tag, asset)` **prefers the `.gz` sibling** and falls back to the raw asset on any non-ok response (a tag published before Stage 19 has no `.gz`), returning `{ res, compressed, url }`. `runUpdate` streams the body into the `.leettui.update.tmp` file — through a `node:zlib` `createGunzip` transform when `compressed` — before the atomic `rename` + `chmod`. The byte-counting progress bar counts on `src` (pre-gunzip), i.e. **compressed** bytes against the compressed `content-length`. GitHub serves `.gz` assets as opaque blobs (no `Content-Encoding: gzip`), so `fetch` won't transparently decompress. A corrupt/truncated gzip stream rejects via the gunzip `error` event → lands in `catch` → `rm(tmp)`, so the temp-then-rename guarantee means the running binary is never clobbered. The `.gz`-preferred / raw-fallback URL resolution is unit-tested in `update.test.ts` (stubbed `fetch` keyed per URL); measured delta: raw **111 MB → 40 MB** (~64% smaller download/update). Presentation goes through `updatePresent.ts`: a branded wordmark banner up top, a live download progress bar (byte counting via a `data` listener on the streamed body — doesn't disturb `pipe` backpressure), and branded `✓`/`✗` success/error lines. For the in-app **update banner**: `checkForUpdate()` returns the newer release **tag** to advertise (or `null`) — fail-silent, `IS_RELEASE`-gated like `runUpdate`, `LEETTUI_FAKE_UPDATE` env override for dev preview; `isNewerVersion(latest, current)` is a pure semver compare (tolerates leading `v`/ignores `-suffix`, `false` on unparseable input so a garbled tag never raises a false banner). The **changelog popup (Stage 18) is a separate, post-update concern**: `fetchLatestRelease()` (`/releases/latest`) and `fetchReleaseByTag(tag)` (`/releases/tags/{tag}`) wrap a shared `fetchRelease`, returning a `ReleaseInfo` `{ tag, body }` (the curated GitHub notes); `releaseUrl(tag)` builds the release-page URL for the popup's `o` (open-on-GitHub) key (single-sources `REPO`). `shouldShowChangelog(currentVersion, lastShownVersion, mode)` is the pure **post-update** decision (unit-tested): `undefined` last-shown → `false` (fresh-install seed case), `currentVersion === lastShownVersion` → `false` (already shown), else `mode === "browse"`. `BootFlow` (official builds only): fetches `fetchReleaseByTag(VERSION)` + opens `ChangelogPopup` **once** when `shouldShowChangelog` passes (re-validating browse mode after the async fetch), then records `setLastShownChangelogVersion(VERSION)` **unconditionally** — which both seeds a fresh install "caught up" (so it never pops on first launch) and marks an update shown, so the popup fires only on the first launch after a genuine version change. The command-palette "What's new" (`handleViewChangelog`) instead shows the **latest** release via the un-gated `fetchLatestRelease` (most useful when behind; works on dev/from-source builds). **Three distinct "told the user" states, kept separate so dismissing one never suppresses the others:** the banner's session-only dismiss (`uiSlice.updateAvailable`), the module-level on-quit reminder (`setPendingUpdate`, read by `index.tsx`'s `process.on("exit")` → `updateNotice(...)` after the renderer tears down), and the **persistent** per-version changelog-shown flag in `session.json`.
- `updatePresent.ts` — Console (renderer-less) presentation for the `update` subcommand and the on-quit "update available" notice; sibling to `cli/present.ts` but kept in `core/` since `update.ts` owns these stdout paths. ANSI color gated on a stdout TTY + `NO_COLOR` (piped `leettui update` stays plain). Exports `brandBanner()` (block-letter wordmark mirroring `onboarding/Logo`, with a narrow single-line fallback), `startProgress(label)` → `{ update(received, total), done(finalLabel?) }` (a TTY carriage-return-redraw bar; off-TTY prints the label once and the bar is a no-op so output stays one line), `updateSuccess`/`updateError` lines, and `updateNotice(tag, current)` (the on-quit reminder).
- `treeSitterWorker.ts` — Side-effect-only module imported **first** in `src/index.tsx`. Points OpenTUI at its tree-sitter highlight worker (`OTUI_TREE_SITTER_WORKER_PATH`) so `<markdown>` is actually syntax-highlighted in the compiled binary. **The build must add `node_modules/@opentui/core/parser.worker.js` as a second `--compile` entry point** (see `scripts/build.ts` and `.github/workflows/release.yml`) — that's what bundles the worker *and its `web-tree-sitter` dependency + engine wasm* into the binary; this module only resolves the embedded path (`/$bunfs/root/node_modules/@opentui/core/parser.worker.js`, computed from `import.meta.url` since Bun flattens module paths in the binary). Without both halves the worker can't load, highlighting silently degrades to one flat unstyled chunk, and `buildMarkdownSyntaxStyle()` does nothing. Two failure modes were seen: pre-fix releases threw `ModuleNotFound parser.worker.ts`; the v0.1.7 attempt (raw `type: "file"` embed of the worker) shipped the worker bytes but not its deps, so it threw `Cannot find package 'web-tree-sitter'` on any machine without a stray `node_modules` beside the binary. Dev (`bun src/index.tsx`) always worked — the worker is a real file on disk — which is why this was release-only.
- `auth/` — Authentication helpers. Two front-ends share them: the in-renderer onboarding `AuthWizard` (`ui/components/onboarding/`) handles first-run/boot auth, and `runAuthFlow()` handles mid-session re-auth on the plain terminal inside a `renderer.suspend()` block.
  - `index.ts` — exports the shared pieces: `validateTokens(csrf, session)` (a cheap `globalData { userStatus }` check via the non-singleton `createClient`, returning `ok`/`invalid`/`unknown`), `openInBrowser`, `LOGIN_URL`, `MAX_PASTE_ATTEMPTS`, and `runAuthFlow()` — the plain-terminal flow (Firefox auto-import → guided paste fallback) used by the Ctrl+P "Re-authenticate" command. Every candidate is checked by `validateTokens` before `persistTokens` saves it, so only working credentials are written.
  - `firefox.ts` — `readFirefoxCookies()` reads `LEETCODE_SESSION`/`csrftoken` from Firefox's plaintext `cookies.sqlite` (copy-to-temp + read-only open to dodge the write lock); scans standard/snap/flatpak/macOS profile roots and keeps the freshest pair that has both tokens.
  - `paste.ts` — `parseCookieInput(raw)`, a pure parser (unit-tested in `paste.test.ts`) accepting a Cookie header, a `document.cookie` dump, or `KEY=VALUE` lines.
