# core/

Business logic that bridges the API, database, and UI layers.

## Files

- `sync.ts` — `syncQuestions(onProgress?)` fetches all LeetCode problems in 1000-item pages, upserts into SQLite within a transaction. `syncIfEmpty()` only syncs if DB has zero questions (first run).
- `search.ts` — `filterQuestions(questions, needle)` performs fuzzy matching against question title and ID. Returns scored and sorted results. Substring matches score highest.
- `solutions.ts` — Solution file manager for the `~/.local/share/leettui/solutions/` directory. **Layout (Stage 7):** `{paddedId}_{slug}/{langSlug}/solution.{ext}`, with shared `notes.md` + `tests/` reserved at the problem-folder level. The **langSlug is a solution's identity** (the directory name) — the filename is always `solution.{ext}`, so callers pass langSlugs, never filenames.
  - `getProblemDir(id, slug)` — `solutions/{paddedId}_{slug}` (no mkdir)
  - `getSolutionPath(id, slug, langSlug)` — `…/{langSlug}/solution.{ext}`; mkdirs the lang subfolder
  - `getNotesPath(id, slug)` / `getTestsDir(id, slug)` — shared problem-level locations
  - `readNotes(id, slug)` → notes.md content or `null` (no side effects); `ensureNotesFile(id, slug, title?)` — mkdir + create-if-absent (seeds a minimal `# {id}. {title}` header on first creation only), returns the path ready for `$EDITOR`. Back the per-problem notes view (Stage 7 item 2); creating `notes.md` never flips the "has solution" marker since `langSlugsWithSolution` only counts lang subdirs
  - `createSolutionFile(id, slug, langSlug, code)` — writes the file only if it doesn't exist
  - `createSolutionWithHarness(id, slug, langSlug, code, metaDataRaw?, exampleTestcases?)` — create-flow orchestrator used by both create call sites (browse `handleOpenEditor`, problem `handlePickerOpenEditor`). Order: (1) **overlay per-language template overrides** (Stage 7 item 6) from `~/.config/leettui/templates/{langSlug}/`, then fill gaps with bundled defaults — (2) the solution file (LeetCode snippet, unless a `solution.{ext}` template supplied one), (3) the generated harness (`main.py`/`main.js`/`main.ts`, unless a same-named template supplied one), (4) seed `tests/` from the example cases. Stays network-free; the handlers fetch `consolePanelConfig` (for `metaData` + `exampleTestcaseList`) and pass the data in
  - **Template overrides** (Stage 7 item 6) — `overlayTemplates(srcDir, destDir, vars)` renders every file in a per-language template dir into the lang folder (create-if-absent, returning the set of filenames it "owns") and `renderTemplate(text, {functionName?, titleSlug})` substitutes `{{functionName}}`/`{{titleSlug}}` (functionName from `metaData`, empty when absent). **Filename convention** (the user-facing contract): a `solution.{ext}` template overrides the LeetCode snippet, a harness-named template (`main.py`/`main.js`/`main.ts`) overrides the generated harness, and **any other filename** (e.g. `Cargo.toml`, `build.rs`, helper modules) lands purely additively. **Top-level files only** — subdirectories in the template dir are not recursed (matches the flat lang-folder model). Overlay runs first; because every default writer is create-if-absent, the ordering is the real safety net and the "applied set" check is belt-and-suspenders. Tests: `solutions.test.ts` (pure `renderTemplate`/`overlayTemplates` in-process; the create-flow end state via a subprocess with a temp `$HOME`, since Bun fixes `os.homedir()` at launch). **`core/harness/index.ts` was deliberately not touched** — routing harness overrides through `generateHarness` would force fs lookups + substitution into its pure, never-throws contract; the unified overlay handles the harness as just another (specially-named) template file
  - `getLangFilePath`/`getHarnessPath`/`createHarnessFile(id, slug, langSlug, filename, content)` — write an arbitrary file inside the lang folder (`getHarnessPath` delegates to `getLangFilePath`); `createHarnessFile` is create-if-absent (never clobbers user edits)
  - `seedTests(id, slug, exampleTestcases)` — writes `tests/case-NN.txt` from LeetCode example cases (each entry written verbatim, create-if-absent; empty list is a no-op)
  - `readSolutionFile(id, slug, langSlug)` — reads solution code (no side effects)
  - `findExistingSolutions(id, slug)` — returns the **langSlugs** that have a `solution.*` file (skips the shared `tests/`/`notes.md`); both this and `listSolutionQuestionIds` share one private `langSlugsWithSolution(dir)` predicate so the marker can never disagree with the problem view
  - `listSolutionQuestionIds()` — returns the set of question IDs whose problem folder contains at least one `solution.*` file (backs the question-list "solution exists" marker); a folder holding only `notes.md`/`tests/` is not counted
- `harness/` — Local test-harness generation (Stage 7 items 3 + 5 + TS). `index.ts` exposes `generateHarness(langSlug, metaDataRaw)` → `{ filename, content } | null`; it dispatches by language (`python3` → `python.ts`, `javascript` → `javascript.ts`, `typescript` → `typescript.ts`) and **never throws** (returns `null` on an unsupported language or missing/unparseable `metaData`, so solution creation can't fail on it). `meta.ts` holds the language-agnostic `MetaData` type + `parseMetaData` + `baseType` + `DEFERRED_TYPES` (the `ListNode`/`TreeNode` set), shared by every generator. `python.ts` (`generatePythonHarness`, unit-tested in `python.test.ts`) emits a `main.py` that reads one JSON value per line from **stdin**, `json.loads`-es each into the positional args, calls `Solution().{fn}(...)`, and prints `json.dumps(result)`. `ecma.ts` holds the shared JS/TS skeleton (`renderArgReads` + `generateEcmaHarness`): both read one JSON value per stdin line, call `{fn}(...)`, and print `JSON.stringify(result ?? null)`, differing only in their `loadSolution()`. `javascript.ts` (`main.js`, run by `node`) loads the function via `new Function(src + "return {fn};")` over the raw `solution.js` text — LeetCode JS snippets are bare un-exported functions and the file must stay submittable, so `require` is out. `typescript.ts` (`main.ts`, run by `bun`) does the same but first strips types with `new Bun.Transpiler({ loader: "ts" }).transformSync(src)` — raw type annotations are invalid JS and would throw `Unexpected token ':'` in `new Function`. Both keep `solution.{js,ts}` byte-for-byte submittable; JS + TS are unit + end-to-end tested (`javascript.test.ts`, `typescript.test.ts` — the TS e2e uses a genuinely *typed* solution so it actually exercises the transpile). `ListNode`/`TreeNode` params are passed through raw with a `// TODO` comment (real deserializers deferred). `index.ts` also exposes `getRunnerSpec(langSlug)` → `{ harnessFilename, command } | null` (the `RUNNERS` registry: `python3 → main.py`/`["python3"]`, `javascript → main.js`/`["node"]`, `typescript → main.ts`/`["bun"]`) — the seam the item-4 runner consults, so a new language is one `case` + one entry and the runner stays unchanged. Each `command` is a toolchain the user needs on PATH (including `bun`: the shipped `--compile` binary does **not** put `bun` on PATH, so it's the same caveat as `node`/`python3`); a missing one degrades to a per-case `error`.
- `testRunner.ts` — Local test runner (Stage 7 item 4). `runLocalTests(question, langSlug)` resolves `getRunnerSpec`, spawns `[...command, harnessFilename]` per case with **cwd = the lang folder** (so the harness imports the sibling `solution.*`) and the `tests/case-NN.txt` file on **stdin**, draining stdout/stderr; a 10s timeout kills runaway cases. Each case gets a `CaseStatus`: `pass`/`fail` (only when an optional sibling `case-NN.out` exists to diff against), `ran` (executed, nothing to compare), `error` (non-zero exit), `timeout`. Returns a `LocalRunReport` discriminated union (`unsupported`/`no-harness`/`no-cases`/`ran`). **Key UX rule:** seeded cases ship inputs only (LeetCode gives no expected outputs), so a run with no `.out` files renders as info ("Ran N cases (no expected outputs)"), never a green pass — `buildLocalRunView` in `views/browse/resultView.ts` derives `kind`/title from whether any case had a verdict. Pure parts (`pairCases`, `compareOutput` — **JSON-normalized**: the harness emits compact `[0,1]` but a copied `.out` uses display spacing `[0, 1]`, so both sides are canonicalized (recursive key sort, float-normalizing) before comparing; falls back to trailing-whitespace/CRLF-insensitive string equality when either side isn't valid JSON) unit-tested in `testRunner.test.ts`.
- `migration.ts` — `migrateSolutionsLayout(dir?)`: one-time, **idempotent**, **collision-safe** migration of legacy flat files (`0001_two-sum.py`) into the folder layout. Skips unknown-extension/non-matching files (loss-free), never overwrites an existing target, and continues past per-file errors. Run once at boot in `BootFlow` after `openDatabase`. Takes an optional dir for testing (`migration.test.ts`).
- `submission.ts` — `runSolution`/`submitSolution` services. On `submit_accepted` also records `last_runtime`/`last_memory` via `setSubmissionStats`.
- `session.ts` — Persists the last-viewed browse position (`{ topicSlug, questionId }`) to `~/.local/share/leettui/session.json`. `loadSession()` on boot; `saveSession()` is debounced (~400ms) because navigation fires per keypress.
- `markdown.ts` — `htmlToMarkdown(html)`: a single shared, configured `node-html-markdown` converter for LeetCode descriptions. Maps `<sup>`/`<sub>` to `^`/`_` (so exponents like `10^5` survive), drops `<img>`, fences code blocks, normalizes `&nbsp;`. Used by both browse and problem handlers.
- `clipboard.ts` — `copyToClipboard(text)` via the OSC 52 terminal escape (works over SSH, no external helper); `problemUrl(slug)` builds the canonical problem URL.
- `treeSitterWorker.ts` — Side-effect-only module imported **first** in `src/index.tsx`. Points OpenTUI at its tree-sitter highlight worker (`OTUI_TREE_SITTER_WORKER_PATH`) so `<markdown>` is actually syntax-highlighted in the compiled binary. **The build must add `node_modules/@opentui/core/parser.worker.js` as a second `--compile` entry point** (see `scripts/build.ts` and `.github/workflows/release.yml`) — that's what bundles the worker *and its `web-tree-sitter` dependency + engine wasm* into the binary; this module only resolves the embedded path (`/$bunfs/root/node_modules/@opentui/core/parser.worker.js`, computed from `import.meta.url` since Bun flattens module paths in the binary). Without both halves the worker can't load, highlighting silently degrades to one flat unstyled chunk, and `buildMarkdownSyntaxStyle()` does nothing. Two failure modes were seen: pre-fix releases threw `ModuleNotFound parser.worker.ts`; the v0.1.7 attempt (raw `type: "file"` embed of the worker) shipped the worker bytes but not its deps, so it threw `Cannot find package 'web-tree-sitter'` on any machine without a stray `node_modules` beside the binary. Dev (`bun src/index.tsx`) always worked — the worker is a real file on disk — which is why this was release-only.
- `auth/` — Authentication helpers. Two front-ends share them: the in-renderer onboarding `AuthWizard` (`ui/components/onboarding/`) handles first-run/boot auth, and `runAuthFlow()` handles mid-session re-auth on the plain terminal inside a `renderer.suspend()` block.
  - `index.ts` — exports the shared pieces: `validateTokens(csrf, session)` (a cheap `globalData { userStatus }` check via the non-singleton `createClient`, returning `ok`/`invalid`/`unknown`), `openInBrowser`, `LOGIN_URL`, `MAX_PASTE_ATTEMPTS`, and `runAuthFlow()` — the plain-terminal flow (Firefox auto-import → guided paste fallback) used by the Ctrl+P "Re-authenticate" command. Every candidate is checked by `validateTokens` before `persistTokens` saves it, so only working credentials are written.
  - `firefox.ts` — `readFirefoxCookies()` reads `LEETCODE_SESSION`/`csrftoken` from Firefox's plaintext `cookies.sqlite` (copy-to-temp + read-only open to dodge the write lock); scans standard/snap/flatpak/macOS profile roots and keeps the freshest pair that has both tokens.
  - `paste.ts` — `parseCookieInput(raw)`, a pure parser (unit-tested in `paste.test.ts`) accepting a Cookie header, a `document.cookie` dump, or `KEY=VALUE` lines.

## Dependencies

- `sync.ts` depends on `api/queries/` and `db/questions`
- `search.ts` depends on `db/questions` (types only)
- `solutions.ts` depends on `config/` (solutions dir + templates dir accessors), `api/types` (language extensions), and `harness/` (`generateHarness`, plus `meta` `parseMetaData` for the template `functionName`)
- `migration.ts` depends on `config/` (solutions dir) and `api/types` (`EXTENSION_TO_LANGSLUG`)
- `auth/` depends on `config/` (`persistTokens`) and `api/client` (`createClient`, `AuthError`); `firefox.ts` uses `bun:sqlite`
