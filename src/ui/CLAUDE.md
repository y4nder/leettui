# ui/

OpenTUI React components, state management, theme, and keymap.

## Files

- `markdownStyle.ts` — `buildMarkdownSyntaxStyle()` maps OpenTUI markdown scopes (`markup.heading`, `markup.strong`, `markup.raw`, `markup.link`, …) onto the active theme tokens so `<markdown>` descriptions are theme-colored. Callers memoize it on `themeVersion` so it rebuilds on theme switch.
- `theme.ts` — Theme controller. Owns the `colors` Proxy (live token reads), `setTheme(name, { persist })`, `cycleTheme(±1)`, `getCurrentThemeName()`, `listThemeNames()`, `rebuildSystemTheme()` (called by `palette.ts` on terminal-palette changes), and helpers `difficultyColor()`, `statusIcon()`, `statusColor()`. Switching a theme bumps `themeVersion` on the store; `BrowseView` and `ProblemView` subscribe to it so their subtrees rerender.
- `themes/` — Preset palettes implementing the `Theme` interface (token values are `string | RGBA`, since OpenTUI's `fg`/`backgroundColor` props accept both). Three presets: `tokyo-night.ts`, `catppuccin.ts` (hex literals), and `system.ts` (RGBA-based — uses `RGBA.defaultForeground()`/`defaultBackground()`/`fromIndex(n)` so colors honor the user's terminal palette). The interface includes both legacy tokens (`bg`/`fg`/`fgAccent`/...) and semantic tokens (`surface`/`subtle`/`accent`/`success`/`warn`/`error`/`info`).
- `palette.ts` — Boot-time `attachPaletteListener(renderer)`. Warms `renderer.getPalette()` to unlock OSC detection, then subscribes to `CliRenderEvents.PALETTE` so the system theme can re-derive when the user changes their terminal colorscheme.
- `keymap/` — Command catalog and binding specs built on `@opentui/keymap`. A **barrel module** (`index.ts` re-exports the whole public surface unchanged, so importers still use `from "../keymap"`); split internally into `command.ts` (`CommandSpec`/`makeCommand`/`ActionCategory`), `commands/{index,browse,problem,modal,system}.ts` (the `COMMANDS` catalog — assembled in `commands/index.ts`, which owns `COMMAND_BY_NAME`/`SHORT_BY_NAME`), `bindings.ts` (per-scope `Binding[]` + `bindingsFor`), `scopes.ts` (`describeScope`/visibility predicates/`footerSegments`), `format.ts` (pure key-string helpers), and `runtime.ts` (`_keymap`/`_renderer` singletons, scroll registries, `installKeymap`). Exports:
  - `installKeymap(keymap, renderer)` — registers every `Command` (with `title`/`category`/`group` metadata) in a global layer and wires the search-mode `key:after` text-input intercept. Called once from `src/index.tsx`.
  - `browseGlobalBindings`, `topicPanelBindings`, `questionPanelBindings`, `popupBindings`, `resultBindings`, `changelogBindings`, `helpBindings`, `debugBindings`, `searchBindings` — `Binding[]` arrays passed to `useBindings` from each scope-owning component. Browse is split lazygit-style (Stage 11): the global layer is always mounted in browse mode, the two panel layers conditionally mount on the store's `focusedPanel`.
  - `getKeymap()` — accessor for code that needs the `Keymap` instance outside React (rare; React code uses `useKeymap()` from `@opentui/keymap/react`).
  Commands with `group: "modal"` are hidden from the command palette and help popup; `group: "debug"` shows only when `LEETTUI_DEBUG=1`.

### components/

All components use OpenTUI JSX intrinsics (`<box>`, `<text>`, `<scrollbox>`, `<markdown>`).

- `TopicList.tsx` — Left panel: scrollable topic list with `►` selection indicator
- `QuestionList.tsx` — Right panel: per-row status icon, solution-exists marker (`◆`, accent color, from `solutionFileIds`), ID, title, acceptance-rate %, difficulty color, and last-accepted runtime. Uses `backgroundColor` for selected row highlight.
- `StatusBar.tsx` — Bottom bar: shows keybinding hints in browse mode, search input in search mode, and the active difficulty filter (e.g. `[Easy]`) when one is set
- `QuestionPopup.tsx` — Centered absolute-positioned modal with `<scrollbox>` + `<markdown>` for problem descriptions. Used by the daily-challenge flow. Builds its `syntaxStyle` from `markdownStyle.ts` (memoized on `themeVersion`). `Enter` opens the question in the full problem view (`popup.enterProblem` → `handleEnterPopupProblem`, using the `popupQuestion` DB row stashed by `showPopup`; if the daily question isn't locally synced it surfaces a "sync with *" info message instead), `Esc` closes, `j/k` scroll.
- `SelectPopup.tsx` — Language selection modal with j/k navigation. Uses `useKeyboard` hook internally.
- `ResultPopup.tsx` — Modal frame around `<ResultBody>` for browse-mode run/submit results.
- `ResultBody.tsx` — Pure rendering of a `ResultView` (metrics, diff, outputs, error, and the local-runner `cases` list — per-case status glyph, expected/actual expanded only on `fail`). The `loading` kind short-circuits to the shared `<Loading>` spinner. Reused by `ResultPopup` and by `ProblemView`'s inline Result panel.
- `NotesPopup.tsx` — Shared, language-agnostic notes view for `ProblemView` (opened with `n`). Renders `notes.md` as scrollable `<markdown>` (placeholder when empty), owns its own `useBindings` layer (`notesBindings`) which — being mounted after `problemGlobalBindings` — shadows problem-view `e`/`escape`/`q` while open: `e` edits `notes.md` in `$EDITOR` (create-if-absent) then refreshes, `Esc`/`q` close. Works with zero solutions (notes are problem-level, not gated on a focused langSlug). Mounted via `{notes && <NotesPopup .../>}` in `ProblemView`.
- `SolutionPickerModal.tsx` — Telescope-style modal for `ProblemView` (opened with `f`). Two-column layout: a list of every available language for the problem (languages with a solution on disk marked with `●`, from the `existing: Set<langSlug>` in picker state), and a syntax-highlighted code preview of the focused entry — file contents for existing solutions, LeetCode starter snippet for languages not yet started. Owns its own `useBindings` layer (`pickerBindings`). Confirming opens the file in `$EDITOR`, creating it from the snippet first if needed.
- `SolutionsPanel.tsx` — Focusable list of the problem's existing solutions (langSlugs) in `ProblemView`'s right column (Stage 12 item 3), replacing the old one-line active-solution strip. The `[2]` panel: while focused, `j/k` cycle the active solution (`moveFocusedSolution` — the one `e`/`R`/`s`/`t` target), `Enter` opens it in `$EDITOR` (reuses `problem.editorOpen`, so `Enter` ≡ `e`), and **`d` deletes** the active solution (Stage 16 — opens the `DeleteSolutionPrompt` confirm; the handler guards an empty list so `d` on "No solution" is a no-op info, since this layer mounts even when empty); the `f` picker still owns creating a new language. Mirrors `QuestionList`'s focus-aware row highlight (fg differentiator), shows "No solution — press f" when empty, and auto-heights so the `flexGrow` Result panel below absorbs the column remainder. Bindings live in `solutionsPanelBindings`, mounted only while the panel is focused (the j/k cycle-vs-scroll layer swap).
- `RelatedPanel.tsx` — Focusable list of a problem's similar questions in `ProblemView`'s right column, below Result (Stage 12 item 4). The `[4]` panel: `j/k` move the cursor (`moveFocusedRelated`), `Enter` navigates-replace to the focused question via `handleEnterRelated` → `handleEnterProblemView(dbQuestion)`. Premium (🔒) and not-locally-synced entries render dimmed and refuse `Enter` (a `setProblemResult` info message says why); the cursor still moves over them. **Windows** like `QuestionList` (a `scrollOffset` that follows `focusedIndex`) since a popular problem can have 10+ similar questions, with a `maxRows` viewport cap computed in `ProblemView` from available height so the list can't starve Result; titles are JS-truncated (with a reserved double-width `🔒` budget) so a long title can't wrap and overlap. `flexShrink:0` (the windowed content is a fixed row count — shrinking would overlap rows; Result absorbs the slack). Bindings live in `relatedPanelBindings`, mounted only while focused.
- `DeleteSolutionPrompt.tsx` — Confirm modal for the irreversible delete-solution action (Stage 16), opened with `d` from the focused Solutions panel. Mounted only while `problem.deleteConfirm` is non-null; names the language folder being removed and reminds that shared `notes.md`/`problem.md`/`tests/` are kept. A **true modal like the help popup** (not `ChangeLocationPrompt`'s `useKeyboard`, which only works in the layer-free `relocate` mode): `ProblemView` gates the global + panel layers off while it's open (`!deleteConfirm`), and it mounts its own `deletePromptBindings` (`y`/`Enter` → `solution.deleteConfirm`, `n`/`Esc`/`q` → `solution.deleteCancel`) — so the destructive keys can't fire behind it and the `d` that opened it can't re-fire. Confirm calls `handleConfirmDeleteSolution` (`deleteSolution` + `refreshProblemSolutions()` re-list/clamp + `refreshSolutionFiles()` marker update); the langSlug is **frozen at open time** in `problemSlice`, not re-read from focus.
- `HelpPopup.tsx` — Focus-aware keybinding list (Stage 11; generalized Stage 12 item 5). A **pure presenter**: props `scopes: {header,bindings}[]` (Local panel first, then Global) + `closeBindings` (the layer to mount, returning to the right mode) + optional `visible` predicate / `title` / `footerHint`. Each scope renders via the keymap's static seam (`describeScope`/`formatKeys`/`visible`, not runtime `getCommandEntries()`). **Browse** (`mode === "help"`, opened with `?`) passes its focused-panel + `browseGlobalBindings` scopes, `helpBindings`, and the default `isScopeEntryVisible` (hides modal commands). **ProblemView** (`problem.help` sub-modal, also `?`) passes `problemPanelBindings(focusedPanel)` + `problemGlobalBindings`, `problemHelpBindings`, and **`isProblemScopeEntryVisible`** — the problem view's commands are all `group:"modal"`, so it needs a filter that *keeps* modal (only debug stays gated on `LEETTUI_DEBUG`); reusing the browse predicate would render the problem help near-empty. While the problem help is open, `ProblemView` gates the global + panel binding layers off (`!help`) so it's a true modal — the solve keys it lists can't fire behind it. Debug entries appear only when `LEETTUI_DEBUG=1`.
- `DebugPopup.tsx` — Live key/error log overlay (debug mode only). Owns `debugBindings` via `useBindings`.
- `ProgressBar.tsx` — One-line re-sync indicator pinned to the bottom of Browse/Problem during a manual re-sync. A `useTick` spinner + the bespoke `smoothBar` (sub-cell smooth fill) once a total is known, `sweepBar` (bouncing block) while indeterminate, plus `pct` + `current/total` counts. Matches the full-screen `SyncStep` so both sync surfaces read alike.
- `Spinner.tsx` — `useTick(intervalMs=80)`, a monotonic frame counter whose timer is local to the caller (only ticks while mounted, i.e. while a sync is in flight). Drives the `sweepBar` phase; the spinner glyph itself now comes from `opentui-spinner`.
- `Loading.tsx` — the single in-flight affordance for the app: the `opentui-spinner` `star` glyph (accent) + a dim label, in a centered row. Used by `ProblemView`'s "Loading problem…" placeholder and by `ResultBody` for the `loading` result kind (the transient "Running…/Submitting…/Still pending…" states), so every "working…" moment reads like the sync screens.
- `progress.ts` — pure, framework-free bar geometry shared by `SyncStep` + `ProgressBar` (unit-tested in `progress.test.ts`): `smoothBar(ratio, width)` → `{filled, empty}` with a fractional trailing block glyph (eighths) for sub-cell precision; `sweepBar(width, tick, blockSize?)` → `{pre, block, post}` for an indeterminate bouncing sweep. Components color the returned runs with theme tokens (accent fill over a dim track).
- `UpdateBanner.tsx` — One-line informational banner at the top of `BrowseView`/`ProblemView` when a newer release exists. Reads `updateAvailable` (set at boot by `core/update`'s `checkForUpdate`, fired from `BootFlow`); renders nothing when null. Informational ("run `leettui update`") with an `x` dismiss key (the `update.dismiss` command, bound in `browseGlobalBindings` + `problemGlobalBindings`, calls `setUpdateAvailable(null)`) that hides it for the session only — it's not persisted, so it reappears next launch if still out of date. Both views subtract one line from `mainHeight` when it's showing.
- `RecentPopup.tsx` — "Recently viewed" history modal (Stage 20), mounted in `BrowseView` only while `mode === "recent"` (opened with `h`). Cloned from `SelectPopup` (owns a local `selectedIndex` + inline `j/k`/`return`/`escape`/`h` `useBindings` layer) but renders `QuestionList`-style rows (status icon · solution marker · id · title · difficulty · a relative "Nh ago" viewed-at stamp via `relativeTime.ts`'s `formatRelative`) from the store's `recents` snapshot (each row a `RecentQuestion` carrying its `viewedAt`), with `QuestionList`'s windowing so the selection stays visible past ~50 rows. `Enter` calls `hideRecent()` then `handleEnterProblemView` (which re-records → bumps to top); empty history shows a placeholder. The snapshot is captured at open time by `handleOpenRecent` (`browse/handlers/daily.ts` → `getRecents()` → `showRecent`), mirroring `showSelect`/`showChangelog`.
- `ChangelogPopup.tsx` — "What's new" popup (Stage 18): a release's notes as scrollable `<markdown>`, mounted in `BrowseView` only while `mode === "changelog"`. A pure presenter (props `{tag, body}`) on the `NotesPopup` pattern — owns its `changelogBindings` layer: `o` opens the full release page on GitHub (`core/auth`'s `openInBrowser` + `core/update`'s `releaseUrl`), `j/k` scroll the shared popup scroller (`registerPopupScroller`), `Esc`/`q` close. **Post-update**: auto-shown once on the first launch after an update — the *running version's* notes (`BootFlow` → `shouldShowChangelog` + `fetchReleaseByTag(VERSION)`, official builds only); the command palette ("What's new", `changelog.open` → `handleViewChangelog`) instead shows the **latest** release (un-gated, works on dev/from-source). The payload lives in `uiSlice.changelog` (a `ReleaseInfo`); the per-version "already shown" flag lives in `core/session`, distinct from the banner's session-only dismiss.
- `onboarding/` — First-run / boot experience, rendered by `BootFlow` before `<App>`. `Logo.tsx` (custom hand-crafted block-letter wordmark with a `reveal` wipe and a narrow single-line fallback), `Splash.tsx` (brief `useTimeline` logo reveal shown every launch, skippable with any key), `AuthWizard.tsx` (in-renderer Firefox-import → guided-paste auth, reusing `core/auth` helpers + `<input>`/`usePaste`), `SyncStep.tsx` (logo + spinner + bespoke `smoothBar`/`sweepBar` driven by `syncSlice` — see `Spinner.tsx`/`progress.ts`), `RelocatePrompt.tsx` (Stage 10 item 2 — branded y/n offer to move already-solved problems when the resolved solutions dir changed; `useKeyboard`, calls `relocateSolutions` then shows the real `{moved, skipped}`), `SolutionsOnboarding.tsx` (Stage 10 item 3 — first-run-only "where should solutions live?" step; `<input value={defaultDir}>` pre-filled with the XDG default, Enter accepts/no-write, a typed path calls `persistSolutionsDir`, Esc accepts default), and `BootFlow.tsx` (the `splash → auth → (solutions?) → loading → (relocate?) → ready` state machine). See `src/CLAUDE.md`.
- `CommandPalette.tsx` — Ctrl+P fuzzy-searchable list of commands sourced from `keymap.getCommandEntries()`. Owns its own `useBindings` layer for navigation and a `key:after` intercept for needle text input. Executes the selected command via `keymap.runCommand(name)`.
- `ChangeLocationPrompt.tsx` — In-TUI "change solutions directory" action (Stage 10 item 4), reached from the command palette (`solutions.changeDir`) → `mode === "relocate"`, rendered by `BrowseView`. A centered modal with three internal phases (`input → confirm → done`, self-contained sub-state like the boot `RelocatePrompt`): pre-filled `<input value={getSolutionsDir()}>`, a `from → to` + count confirm, then the real `{moved, skipped}` result. **Works alongside the live keymap because no key-bearing layer mounts in `"relocate"` mode** (browse/search/palette bindings all gate on their own mode; the search `key:after` intercept checks `mode === "search"`), so every key falls through to the `<input>`/`useKeyboard`. The persist+migrate is the gated `changeSolutionsDir` handler (`views/browse/handlers/location.ts`); the component only calls `refreshSolutionFiles()` afterward.
- `EasterEgg.tsx` — Full-screen ASCII logo reveal reached from the command palette (`egg.splash` → "✦ Reveal the leettui logo"). Mounted only in `easterEgg` mode, so no keymap bindings are active; it dismisses on **any** key via `useKeyboard` → `hideEasterEgg()`. Reuses `onboarding/Logo` and the `useTimeline` reveal.

### store/

Zustand store. **UI vs domain rule:** every slice file starts with a `// type: ui` or `// type: domain` header. UI slices own cursors, modes, and ephemeral display state. Domain slices own data sourced from DB/API. A UI slice may call domain actions; a domain slice never imports UI state. If a new slice would mix both, split it.

- `questionsSlice.ts` (**domain**) — `allTopics` (full DB list) + `topics` (displayed projection, filtered by the topics-panel search — mirrors `allQuestions`/`filteredQuestions`), `allQuestions`, `filteredQuestions`, `stats`, `solutionFileIds` (set of question IDs with a solution file). Actions: `init()`, `refreshQuestions()`, `refreshSolutionFiles()`, `loadTopic(topic)`, `applySearch(needle)`, `applyTopicSearch(needle)` (fuzzy-filter `topics` from `allTopics` synchronously so the list tracks each keystroke; empty needle restores full list + leaves questions; a real needle filters + **debounce-loads** the top match's questions via `scheduleTopicLoad`, synchronous branches `cancelTopicLoad`), `applyFilters()`. Owns DB reads and the `project(all, needle, difficulty)` filter pipeline (difficulty predicate then fuzzy search).
- `selectionSlice.ts` (**ui**) — `selectedTopicIndex`, `selectedQuestionIndex`. Actions: `moveQuestion()`, `setQuestionIndex()` (backs gg/G), `moveTopic()`, `setTopicIndex()`, `restoreSession()`. Topic moves mutate the cursor instantly but **debounce** the domain `loadTopic` via `store/topicLoad.ts` (so holding j/k doesn't fire a query per keypress); they persist `topicSlug` immediately and `questionId` only after the deferred load settles. Mirrors the cursor to disk (debounced) via `core/session`.
- `searchSlice.ts` (**ui**) — **panel-scoped search** (Stage 11 item 8): `searchNeedle` (questions) + `topicNeedle` (topics). The active needle is keyed off `focusedPanel`, which is frozen during search mode (the focus keys live in `browseGlobalBindings`, unmounted while searching). Actions: `startSearch()` (scopes to the focused panel; topic search expands to the full list first), `updateSearch()` (routes to `applySearch`/`applyTopicSearch`, resetting the right cursor — incl. `selectedQuestionIndex` on a topic reload), `endSearch()` (both panels persist their filtered projection). A private `restoreFullTopics()` remaps the highlighted topic's slug into `allTopics` so a bare `/` neither jumps to "all" nor reloads.
- `filtersSlice.ts` (**ui**) — `difficultyFilter` (`all`/`Easy`/`Medium`/`Hard`). Action: `cycleDifficulty()` (bound to `D`); applied alongside the search needle by the domain slice's `applyFilters()`.
- `uiSlice.ts` (**ui**) — Browse-side UI + generic modals. `mode` (`AppMode`, **co-owned with `problemSlice`** — its problem actions also write `mode`, legal since `set` is typed over the whole `AppStore`), `focusedPanel` (`"topics" | "questions"` — lazygit-style browse focus, Stage 11; `setFocusedPanel`/`cycleFocusedPanel`), popup/select/result modal content with show/hide actions (`showPopup`, `hidePopup`, `hideResult`, `hideHelp`, …; `showPopup` also stashes the optional `popupQuestion` DB row so the daily popup's `Enter` can open it in the problem view), `themeVersion`, `updateAvailable` (the newer release tag for the top banner, or null — set by `BootFlow`'s `checkForUpdate`), `changelog` (the `ReleaseInfo` `{tag, body}` whose "What's new" popup is open, or null — `showChangelog`/`hideChangelog`, Stage 18), and `recents` (the recently-viewed `DbQuestion[]` snapshot backing the `h` history modal, captured at open time — `showRecent`/`hideRecent`, Stage 20). `BrowsePanel`/`PANEL_ORDER` live here.
- `problemSlice.ts` (**ui**) — The entire **ProblemView domain**, extracted from `uiSlice` (P2 refactor; the two slices co-own `mode`). `problem: ProblemViewState | null` (active question, description, `solutions` list — **langSlugs**, not filenames — focused index, `related` list of similar questions + `focusedRelatedIndex` (Stage 12 item 4 — each `RelatedQuestion` carries its resolved `dbQuestion` so navigability is known offline), **`focusedPanel: ProblemPanel`** (`"description" | "solutions" | "result" | "related"` — ProblemView's own lazygit-style focus, Stage 12; sibling to the browse `focusedPanel`, drives the j/k target — scroll for description/result, active-solution cycle for Solutions, cursor move for Related — plus the accent border via `PROBLEM_PANEL_ORDER`), inline result, modal solution-picker state whose `existing` is a `Set<langSlug>`). Actions: `enterProblemView` (takes the fetched `related`; writes `mode`), `exitProblemView`, `setProblemSolutions`, `moveFocusedSolution`, `moveFocusedRelated`, `setProblemResult`, `setProblemFocusedPanel`/`cycleProblemFocusedPanel` (linear, Tab) / `moveProblemFocus(dir)` (spatial, Ctrl+h/j/k/l — consults a `PROBLEM_PANEL_NEIGHBORS` map, no-op at edges), `openSolutionPicker`, `movePicker`, `closeSolutionPicker`, and the problem sub-modals `notes` / `help` / `deleteConfirm` (`openNotes`/`closeNotes`, `openProblemHelp`/`closeProblemHelp`, `openDeleteConfirm`/`closeDeleteConfirm` — all nested in `problem` so they close back to the problem view, not browse; `deleteConfirm` (Stage 16) carries the `{ langSlug }` frozen at open time so a focus change behind the modal can't retarget the irreversible delete). Owns/exports `ProblemViewState`, `RelatedQuestion`, `SolutionPickerState`, `ProblemPanel`, `FocusDirection`, `PROBLEM_PANEL_ORDER`, `PROBLEM_PANEL_NEIGHBORS`.
- `syncSlice.ts` (**ui**) — `syncProgress`. Actions: `setSyncProgress()`, `clearSyncProgress()`.
- `index.ts` — Combines slices into `useAppStore`. Re-exports `AppMode`/`BrowsePanel`/`PANEL_ORDER` (from `uiSlice`) and the problem types `ProblemPanel`/`RelatedQuestion`/`PROBLEM_PANEL_ORDER` (from `problemSlice`) — external callers import these from this barrel, so the P2 slice split is invisible to them.
- `topicLoad.ts` — Shared **debounced** topic-load (`scheduleTopicLoad(get, persist?)` / `cancelTopicLoad()`) used by `selectionSlice` (topic j/k) and `questionsSlice` (`applyTopicSearch`). Reset-on-each-call (~90ms) so rapid navigation/typing fires one query on settle, not per keypress; loads the *settled* `topics[selectedTopicIndex]` at fire time. Distinct from `core/session`'s throttle — see the file header for why a throttle would load the wrong (intermediate) topic.

## OpenTUI prop notes

- `<box>` uses `backgroundColor` (not `bg`)
- `<text>` uses `fg` and `bg`
- `<markdown>` requires `content` prop (string) and `syntaxStyle` prop (`SyntaxStyle` instance)
- `<scrollbox>` wraps children for scrollable overflow

## Dependencies

- Components depend on `db/`, `api/`, `core/`, `config/` for data fetching and business logic
- `views/browse/handlers/` and `views/problem/handlers.ts` call `core/markdown.ts`'s `htmlToMarkdown()` (a shared, configured `node-html-markdown` converter) before passing content to `QuestionPopup` / `ProblemView`
